# ETL Platform 고도화 분석

> 프로그램 로직·구현 기능 전체 판단 및 tMap 컴포넌트 기능 분석·업그레이드 방향을 정리한 문서.

---

## 1. 전체 프로그램 로직·구현 기능 요약

### 1.1 아키텍처

- **역할**: Talend Open Studio 스타일의 웹 기반 비주얼 ETL.
- **흐름**: DAG 캔버스(노드/엣지) → **IR(JobIR)** → **SQL Pushdown 컴파일** → 대상 DB에서 `INSERT...SELECT` 실행.
- **스택**: Backend Spring Boot(Kotlin), Frontend React + Vite + TS, React Flow 12, Zustand, 메타 DB PostgreSQL(Docker 5433).

## 2. tMap 컴포넌트 현재 기능 분석

### 2.1 프론트엔드

| 구분 | 내용 |
|------|------|
| **MappingEditorModal** | tMap 더블클릭으로 오픈. **좌측**: ROW로 연결된 Input 노드들의 소스 컬럼(노드별 색상). **우측**: Output 탭(연결된 T_JDBC_OUTPUT별), Source Column / Target Name / Expression / Type 행 편집. 소스 클릭 → 타겟 행 클릭으로 매핑, Auto Map, Clear, Apply 시 `outputMappings[outputId]` 저장. |
| **PropertiesPanel (MapConfig)** | Output별 아코디언(테이블명, writeMode, 매핑 수), Output별 JSON textarea, "시각적 매핑 에디터" 버튼 → 모달 오픈 시 `initialOutputNodeId` 전달. |
| **매핑 데이터** | `MappingRow`: id, sourceNodeId, sourceColumn, targetName, expression, type. 저장은 `config.outputMappings[outputNodeId]`(배열), legacy로 단일 `config.mappings`. |
| **Auto Map** | `buildAutoMappings` / `getAutoExpression`: 타겟 테이블 컬럼 또는 소스 전부에 대해 TRIM, UPPER(TRIM), COALESCE(..., 0), CAST(... AS DATE), 타입별 규칙 적용. |

### 2.2 백엔드

| 구분 | 내용 |
|------|------|
| **SqlPushdownCompiler** | Output 노드별로 컴파일 시, T_MAP에는 `outputMappings[targetOutputNode.id]`가 있으면 그걸 `mappings`로 사용. `parseMappings(mappings)`로 `MappingEntry(sourceColumn, targetName, expression)` 리스트 생성. CTE: `SELECT (expression.ifBlank { sourceColumn }) AS targetName, ... FROM prev`. **sourceNodeId는 사용하지 않음** — 컬럼만으로만 참조. |
| **SqlPushdownAdapter.validate** | tMap에 대해 `config["mappings"]`만 사용하고, `findDownstreamOutput(mapNode.id)`로 **첫 번째로 찾은** Output 노드 1개만 사용. `outputMappings`를 돌며 **각 Output별**로 타겟 컬럼 검사하지 않음. |

### 2.3 현재 한계 요약

1. **다중 Output 검증**: 실제 컴파일은 Output별 `outputMappings[id]`를 쓰는데, 검증은 legacy `mappings` + 첫 번째 Output만 검사 → 다중 Output일 때 나머지 경로 검증 누락.
2. **다중 Input(JOIN) 후 tMap**: 상위 CTE가 `l.*, r.*` 형태여도, T_MAP의 expression은 **컬럼명만** 사용. `sourceNodeId`는 IR에 있지만 컴파일러에서 미사용 → 동명 컬럼이 있으면 애매해질 수 있음.
3. **Expression 검증**: 문법/참조 검사 없음 (구현계획기능 2-1에서 제안된 부분).
4. **Lookup/Reject**: Talend tMap의 Lookup, Reject 출력 포트는 미구현(Reject는 IR만 정의).

## 3. tMap 업그레이드 제안 (기능 확장 방향)

### 3.1 우선 적용 권장 (호환·검증 보강)

| 항목 | 내용 |
|------|------|
| **다중 Output 검증** | `SqlPushdownAdapter.validate()`에서 tMap 노드마다 `config["outputMappings"]`를 순회하고, 각 Output 노드 ID에 대해 해당 매핑 배열 + 그 Output의 `config["columns"]`로 타겟 컬럼 존재 여부 검사. `outputMappings`가 없을 때만 기존처럼 `config["mappings"]` + `findDownstreamOutput` 한 번만 검사. |
| **Expression 실시간 검증(1차)** | 구현계획기능 2-1: 정규식/단순 규칙으로 `context.변수명`, 허용 함수/연산자 패턴 검사. MappingEditorModal의 Expression 입력 옆에 검증 결과(에러/경고) 표시. 필요 시 백엔드에 `validateExpression(irJson, nodeId, expression)` 추가. |

### 3.2 중기 (다중 소스·UX)

| 항목 | 내용 |
|------|------|
| **다중 Input 시 컬럼 참조 명확화** | T_MAP 직전이 T_JOIN 등이면 CTE에 `l`, `r` 같은 별칭이 있음. `parseMappings` 또는 `buildCteSql` 단계에서 `sourceNodeId` → upstream CTE 별칭 매핑을 두고, expression이 비어 있을 때 `"별칭.컬럼"` 형태로 생성하거나, 사용자 expression에서 `소스노드라벨.컬럼` 형태를 허용 후 컴파일 시 해당 별칭으로 치환. |
| **상수/표현식 전용 행** | 이미 expression에 `CURRENT_TIMESTAMP`, `context.변수` 등을 넣을 수 있음. UI에서 "소스 없음" 행을 더 명시적으로 지원(예: Source Column "(상수)" 또는 비움 + Expression 필수). |

### 3.3 장기 (Talend 기능 근접)

| 항목 | 내용 |
|------|------|
| **Expression 고급 검증** | JSQLParser 등으로 표현식 파싱 후 컬럼/함수 유효성 검사 (구현계획기능 2-1 후속). |
| **REJECT 출력** | tMap에서 Reject 포트 타입 정의 후, 실행 엔진에서 조건 불일치 행을 REJECT 엣지로 보내는 시맨틱(임시 테이블/파일 등) 설계. |

## 4. 정리

- **전체**: IR 기반 SQL Pushdown + 스케줄/실행 이력/Context/AI까지 구현된 상태이고, tMap은 **다중 Output 매핑(출력별 outputMappings) + 시각적 에디터 + Auto Map**까지 되어 있음.
- **tMap**: 컴파일·저장 구조는 다중 Output을 지원하나, **검증은 단일 Output 가정**이라 다중 Output 검증 보강이 가장 시급하고, 이어서 Expression 1차 검증, 다중 Input(JOIN) 시 소스 구분 보강을 추천함.
