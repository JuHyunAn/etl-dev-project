> WORK_HISTORY.md 및 제안서(미구현 및 핵심 보완 기능)를 기준으로, **현재 코드베이스**와 대조하여 각 제안의 **구현 가능성**과 **기존 구현 로직에의 적용·이식 가능성**을 정리한 문서입니다.  
> v2: 리스크·구체 대안·구현 복잡도·실행 단계 보강. 과도한 낙관/세부 리스크 미지적/복잡도 과소평가를 줄이기 위해 수정됨.

## 답변 시 참고 기준 (재검토 최소화용)

이 문서를 참고해 제안·피드백을 할 때 아래를 적용하면 재고 요청을 줄일 수 있습니다.

- **리스크 명시**: 각 항목별 리스크 수준(저/중/고)과 구체적 위험(보안, 성능, 법적)을 함께 적기.
- **숫자/벤치마크**: 가능하면 지연 시간, 오버헤드(예: CPU %), 스케일 기준(예: 1000+ Job 시)을 예시로 제시.
- **구체적 대안**: “Elasticsearch”만 말하지 않고, 소규모일 때는 “PostgreSQL full-text/tsvector” 같은 대안을 함께 제시.
- **실행 단계**: “이식 가능” 수준에서 그치지 않고, (1)(2)(3) 단계나 예시 코드 스니펫으로 구현 플로우를 구체화.
- **복잡도/한계**: Python 브릿지·동적 워크플로우 등은 “가능”이어도 오버헤드·무한 루프 위험 등 한계를 명시.
- **예상 공수**: 권장 순서에 “예상 구현 시간(일)” 추정을 넣어 로드맵 신뢰도 확보.

## 분석 기준 요약

| 항목          | 현재 구현 상태 (코드 기준)                                                                                                                                                                  |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **실행 엔진** | `ExecutionEngine` 인터페이스 + `SqlPushdownAdapter` 단일 구현. `ExecutionService`에서 `ir.engineType`으로 엔진 선택, `validate()` → `execute()` 호출.                                       |
| **IR**        | `JobIR`(nodes, edges, context), `NodeIR`(config 맵), `ComponentType` enum에 T_FILE_INPUT/OUTPUT, T_RUN_JOB, T_VALIDATE, T_PROFILE, T_LINEAGE 포함. `PortIR`에 이미 `REJECT` 포트 타입 정의. |
| **잡/메타**   | `Job` 엔티티: `version`, `status`(DRAFT/PUBLISHED/ARCHIVED), `irJson`(JSONB). `executions` 테이블 존재하나 **실행 결과 저장 로직 없음**(Controller가 결과만 반환).                          |
| **커넥션**    | `Connection` 엔티티(DB 타입별 host/port/database/schema). **풀링/Repository 없음** — 매 요청 `DriverManager.getConnection`.                                                                 |
| **파일 I/O**  | 팔레트·타입·노드 색상에 T_FILE_INPUT/OUTPUT 존재. **속성 패널·런타임 처리 없음.**                                                                                                           |

## 이미 구현된 항목 vs 스택 부재·사용 가능성 정리

제안 기능 중 **이미 프로그램 로직에 존재하는 항목**과 **현재 기술 스택에 없는 항목**(사용 가능성·적용 순서 정리)을 구분해, 앞으로 어떻게 적용해 나갈지 명시합니다.

### 이미 구현되어 있는 항목 (확장·연결만 하면 되는 것)

| 항목                                         | 현재 구현 위치                                                                                                        | 앞으로 적용 방향                                                                                                                                                                                           |
| -------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **DAG 순환 검사 (CycleDetector)**            | `SqlPushdownAdapter.hasCycle(ir)` — DFS 기반으로 이미 구현됨.                                                         | 고급 워크플로우(루프) 도입 시 **“허용된 루프”**만 예외로 두고, 기존 `hasCycle`은 그대로 활용하거나 `validate()` 단계에서 “루프 노드 + maxIterations” 검사만 추가. **새로 CycleDetector를 만들 필요 없음.** |
| **위상 정렬 (실행 순서)**                    | `ExecutionService.topologicalSort(ir)` — 노드 실행 순서 계산.                                                         | 조건 분기/루프 시 **실행 경로가 동적으로 바뀌는 부분**만 확장(예: 다음 노드 선택 로직). 정적 DAG 정렬은 그대로 사용.                                                                                       |
| **실행 결과 모델 (노드별 결과·에러)**        | `ExecutionModels.kt` — `NodeResult`(rowsProcessed, rowsRejected, errorMessage), `ExecutionResult`, `ExecutionStatus`. | 실행 이력 저장 시 **이 모델을 그대로** `executions` 테이블(또는 node_results JSONB)에 직렬화해 저장. 새 모델 정의 최소화.                                                                                  |
| **executions 테이블**                        | `V3__create_executions.sql` — job_id, status, triggered_by, logs, node_results 등 컬럼 존재.                          | **저장 로직만 추가** (Execution 엔티티·Repository·실행 후 save). 스키마 변경 없이 진행 가능.                                                                                                               |
| **Job.version, Job.status, folders**         | `Job` 엔티티·`V2` 마이그레이션.                                                                                       | 버전 이력은 `job_versions` 테이블 추가 또는 기존 version 필드 정책으로 확장. 폴더는 UI만 연결.                                                                                                             |
| **REJECT 포트 타입·NodeResult.rowsRejected** | `JobIR.kt`(PortType.REJECT, LinkType.REJECT), `ExecutionModels.kt`.                                                   | REJECT 라인 **실행 시맨틱**만 구현(임시 테이블/파일 쓰기). 데이터 구조는 그대로 사용.                                                                                                                      |
| **Preview Mode·previewIr API**               | `ExecutionController.previewIr`, `ExecutionService.previewIr`, `ExecutionRequest.previewMode`.                        | 미리보기 결과 테이블 UI는 이 API 응답을 그대로 활용.                                                                                                                                                       |

| 문서에서 언급된 기술                      | 현재 상태 (build.gradle.kts / package.json 기준)              | 앞으로 적용 방향                                                                                                                                                                                                                                          |
| ----------------------------------------- | ------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

| **JSQLParser**                            | 백엔드 의존성 없음.                                           | Expression 유효성 검사 도입 시 `implementation("com.github.jsqlparser:jsqlparser")` 추가. 또는 정규식/단순 DSL로 1차 검증 후, 필요 시 JSQLParser로 교체하는 **단계적 도입** 가능.                                                                         |
| **Kotest**                                | 백엔드 테스트는 `spring-boot-starter-test`(JUnit)만 사용.     | **현재 스택**: JUnit만 사용 중. Kotest 도입 시 `testImplementation("io.kotest:kotest-runner-junit5")` 등 추가. 우선 JUnit으로 검증 로직 작성 후, 팀 convention에 맞춰 Kotest 전환 여부 결정.                                                              |
| **Jest / Vitest (프론트 테스트)**         | `package.json`에 test 스크립트·테스트 라이브러리 없음.        | **앞으로 적용**: (1) **Vitest 1순위** — 현재 **Vite** 사용 중이므로 Vitest가 설정·툴체인 공유로 프로젝트에 최적. `npm i -D vitest`, `scripts.test` 추가. (2) Jest는 Vitest 도입이 어려울 때만 대안 검토. (3) CI에서 `npm test` 실행 가능하도록 구축 선행. |
| **Ag-Grid**                               | 프론트 의존성 없음.                                           | 미리보기 결과 테이블은 **1차**로 기본 HTML 테이블 또는 간단한 스크롤 리스트로 구현 가능. 대량 행·가상 스크롤이 필요해지면 `ag-grid-react` 등 도입.                                                                                                        |
| **Spring Security (인증/인가)**           | `spring-security-crypto`만 사용(비밀번호 암호화).             | 인증/RBAC 도입 시 `spring-boot-starter-security` 추가. 기존 API 경로는 유지하고, 필터/메서드 시큐리티로 보호 계층만 추가하는 방식으로 적용.                                                                                                               |
| **Micrometer / Actuator**                 | 백엔드 의존성 없음.                                           | 모니터링 도입 시 `spring-boot-starter-actuator`, `micrometer-registry-prometheus` 등 추가. 오버헤드를 줄이려면 **도입 후** `@Observed` 등으로 수집 구간 한정.                                                                                             |
| **Guava RateLimiter / Bucket4j**          | 백엔드 의존성 없음.                                           | AI 프록시 rate limiting 시 **Resilience4j 1순위** 권장(Spring 생태계). Guava/Bucket4j는 대안. in-memory 카운터로 1차 적용도 가능(하단 「프로젝트 성격 기준 스택 최적화 검토」 참고).                                                                      |
| **HikariCP**                              | Spring Data JPA 사용 시 기본 풀. **메타 DB**에는 이미 적용됨. | **대상 DB(ETL 타겟)**용 풀은 **별도** 구성. `DataSource` 빈을 Connection별로 생성할 때 HikariConfig 사용하면 됨. 추가 의존성 없이 표준 Spring/JDBC로 적용 가능.                                                                                           |
| **Elasticsearch / ELK**                   | 스택에 없음.                                                  | 초기에는 **PostgreSQL**(executions 확장·full-text)만 사용. Job/로그 규모가 커지거나 검색 요구가 높아질 때 Elasticsearch·ELK 도입. **단계적 도입**으로 비용·복잡도 제어.                                                                                   |
| **HashiCorp Vault / Spring Cloud Config** | 스택에 없음.                                                  | AI 키·DB 비밀 등은 **1차**로 환경 변수로 적용. 운영 안정화 단계에서 Vault 또는 Spring Cloud Config 도입.                                                                                                                                                  |

### 1-1. 파일 커넥터 (T_FILE_INPUT / T_FILE_OUTPUT)

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --- | ---------------------------------------------------- |
| **구현 가능성**         | ✅ **가능**. 스택(Spring Boot, React)만으로 구현 가능.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **현재 로직 적용·이식** | ✅ **적용 가능**. `ComponentType.T_FILE_INPUT`, `T_FILE_OUTPUT` 이미 IR·팔레트·CustomNodes에 정의돼 있음. `PropertiesPanel`에 `JdbcInputConfig`/`JdbcOutputConfig` 패턴 그대로 따라 파일 경로·포맷(CSV/JSON 등)·헤더 옵션 UI 추가 가능. **실행 경로**: `SqlPushdownAdapter`는 JDBC만 처리하므로, 파일 컴포넌트는 별도 **FileAdapter**(Kotlin/JVM) 구현 후 `ExecutionService`에서 노드 타입별 엔진 분기. **프로젝트 성격**: Plan_ETL “Execution Plane: Java/Kotlin 런타임 + 파일 IO”에 맞추어 **파일 런타임은 JVM(Kotlin) 1순위**. Python Worker는 보조·선택 사항(하단 「프로젝트 성격 기준 스택 최적화 검토」 참고). |
| **리스크**              | **중**. 로컬 파일 경로는 서버 디렉토리 오버플로우·경로 조작 시 보안/법적(GDPR) 이슈 가능. SaaS 배포 시 반드시 대안 적용.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **구체적 대안 (SaaS)**  | (1) **파일 업로드 흐름**: 프론트에서 “업로드 요청” → 백엔드가 S3/GCS **presigned URL** 생성 후 반환 → 클라이언트가 해당 URL로 **직접 업로드**(서버를 경유하지 않음). (2) **Connection 엔티티 확장**: `storageType: LOCAL                                                                                                                                                                                                                                                                                                                                                                                             | S3  | GCS` 필드 추가해 스토리지별 설정(버킷, prefix) 저장. |

### 1-2. 확장 커넥터 (NoSQL, API, 빅데이터)

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구현 가능성**         | ✅ **가능**. 각 소스별 SDK(예: MongoDB, REST 클라이언트) 의존성 추가로 구현 가능.                                                                                                                                                                                                                                                                                                                                                                                                            |
| **현재 로직 적용·이식** | ✅ **이식 가능**. 현재 `ConnectionService`는 JDBC 전제(`buildJdbcUrl`, `DriverManager`). **어댑터 패턴** 도입 시: (1) `Connection`에 `connectionType`(JDBC / FILE / API / MONGO 등) 또는 기존 `dbType` 확장, (2) `SchemaService`처럼 “메타데이터 조회”를 인터페이스로 두고 DB별·API별 구현체 주입, (3) 프론트 Connection 생성 UI에 타입 드롭다운 확장. **이식 시 주의**: 스키마/테이블 개념이 없는 소스(API, NoSQL)는 `listTables`/`getColumns` 의미를 “리소스/컬럼 목록”으로 재해석해야 함. |

### 1-3. 자동 스키마 변경 탐지

| 항목                    | 내용                                                                                                                                                                                                                                                                                              |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구현 가능성**         | ✅ **가능**. Quartz 또는 Spring Scheduled + DB 메타데이터 해시/버전 저장으로 구현 가능.                                                                                                                                                                                                           |
| **현재 로직 적용·이식** | ⚠️ **부분 이식**. 현재는 “연결 시점” 스키마 조회만 있음. 주기적 폴링·알림은 **신규 모듈**(스케줄러, 스키마 스냅샷 테이블, 알림 채널)로 추가. WebSocket은 `CorsConfig`·프론트와 연동해 실시간 푸시 가능. 기존 `SchemaController`/`SchemaService`는 그대로 두고 “스냅샷 비교” 서비스만 추가하면 됨. |

### 1-4. 연결 풀링 및 모니터링

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                                                                                       |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **구현 가능성**         | ✅ **가능**. HikariCP는 Spring Data JPA 기본 풀(메타 DB용)로 이미 사용 중일 가능성 높음. **대상 DB(ETL 타겟)**용 풀을 별도로 두려면 `Connection`별 DataSource 빈 또는 풀 매니저 도입 필요.                                                                                                                                                                                                 |
| **현재 로직 적용·이식** | ✅ **이식 가능**. 현재 `SchemaService`·`SqlPushdownAdapter`는 `ConnectionService.get` + `buildJdbcUrl` + `DriverManager.getConnection`으로 **매 호출 새 연결**. 여기를 “Connection ID → DataSource(풀)” 캐시로 교체하면 됨. **성능 영향 예시**: DriverManager 방식은 연결당 **평균 50~200ms** 지연, HikariCP 풀 hit 시 **1~5ms** 수준 — 스키마 조회·실행 빈도가 높으면 우선순위 상향 권장. |
| **구체적 이식 단계**    | (1) `ConnectionService`에 `Map<UUID, DataSource>` 캐시 추가. (2) 연결 필요 시 캐시 없으면 HikariDataSource 생성 후 캐시에 넣기(필요 시 `@PostConstruct`에서 기존 Connection 목록으로 워밍업). (3) Connection 수정/삭제 시 해당 ID 캐시 **invalidate** 후 풀 종료.                                                                                                                          |

## 2. 데이터 변환 로직 지원 (Transformation Logic)

### 2-1. Expression 유효성 검사

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구현 가능성**         | ✅ **가능**. JSQLParser 등 SQL 파서 또는 정규식/DSL 파서로 표현식 검증 가능.                                                                                                                                                                                                                                                                                                             |
| **현재 로직 적용·이식** | ✅ **적용 가능**. tMap/필터의 `config`(mappings, condition)는 이미 IR에 저장됨. **백엔드**: `ExecutionService`에 `validateExpression(irJson, nodeId, expression)` 같은 메서드 추가하거나, `ExecutionEngine.validate()` 확장으로 “표현식 문법 검사” 항목 추가. **프론트**: PropertiesPanel/MappingEditorModal에서 API 호출 또는 클라이언트 파서로 실시간 검증 후 Monaco/인라인 에러 표시. |
| **스택 반영**           | **JSQLParser**는 현재 `build.gradle.kts`에 없음. 도입 시 의존성 추가 필요. 1차는 정규식/단순 규칙으로 검증 후, 필요 시 JSQLParser 단계적 도입(상단 「현재 스택에 없거나 사용 가능성 정리」 참고).                                                                                                                                                                                        |

| **구현 가능성**         | ✅ **가능**. Kotlin/Java 라이브러리 또는 Python 브릿지로 구현 가능.                                                                                                                                                                                                               |

### 2-3. Pushdown 최적화 (SqlPushdownAdapter 완성)

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구현 가능성**         | ✅ **가능**. IR → Dialect별 SQL 생성기는 설계상 핵심이므로 구현 가능.                                                                                                                                                                                                                                                                                                                           |
| **현재 로직 적용·이식** | ✅ **직접 적용 대상**. 현재 `SqlPushdownAdapter`는 노드별로 **단순 SELECT/실행**만 하고, CTE/전체 파이프라인 SQL 컴파일은 미구현. Plan_ETL/Main_Plan의 “IR → SQL AST → Dialect SQL” 레이어를 여기에 넣으면 됨. JOIN/FILTER를 소스 쿼리로 흡수하는 최적화는 이 컴파일러 단계에서 처리 가능. EXPLAIN/튜닝 UI는 실행 결과에 `generatedSql`이 이미 있으므로, 별도 “설계 단계 검증” API로 확장 가능. |

### 2-4. 멀티 아웃풋 포트 (REJECT 라인)

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **구현 가능성**         | ✅ **가능**. React Flow handle 추가 + IR 확장.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **현재 로직 적용·이식** | ✅ **이식 가능**. `PortType.REJECT`, `LinkType.REJECT`가 이미 `JobIR.kt`에 정의돼 있음. `NodeResult`에 `rowsRejected` 필드 존재.                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **구체적 구현 플로우**  | (1) **IR 파싱**: 실행 계획 빌드 시 outputPorts 중 `portType == REJECT`인 엣지 탐지. (2) **실행 시**: `SqlPushdownAdapter.executeNode`에서 변환/필터 노드 처리 시 **TRY-CATCH**로 오류 행 캡처 → **임시 테이블 INSERT** (예: `jdbcTemplate.batchUpdate("INSERT INTO temp_reject_... VALUES (...)")`) 또는 reject 전용 파일 쓰기. (3) reject 타겟 노드가 있으면 해당 노드(예: T_FILE_OUTPUT)로 “reject 스트림” 전달 — 단, 현재 엔진이 행 단위가 아니라 SQL 푸시다운이면 “실패 시 전체 롤백 vs. reject 테이블에만 적재” 정책을 IR 또는 노드 config에 명시 필요. |

### 3-2. 고급 워크플로우 (조건 분기, 루프)

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구현 가능성**         | ✅ **가능**. DAG 확장(조건 엣지, 루프 메타노드)으로 표현 가능.                                                                                                                                                                                                                                                                                                                                           |
| **현재 로직 적용·이식** | ⚠️ **부분 이식**. 현재 IR은 **단순 DAG**(위상 정렬로 순서만 결정). 분기/루프는 (1) IR에 “조건식” 엣지 또는 “서브그래프” 노드 추가, (2) `ExecutionService`에서 “조건 평가 후 다음 노드 선택”으로 실행 경로 동적 결정.                                                                                                                                                                                     |
| **리스크·복잡도**       | **중~고**. 동적 경로/루프는 **무한 루프·스택 오버플로우** 위험이 있으며, 실무에서 워크플로우 실패 원인의 상당 비중을 차지. **필수 보완**: (1) IR에 루프 노드용 **`maxIterations`** 필드 추가(기본값 예: 100), 초과 시 실패 처리. (2) 순환 참조 검사는 **이미 구현됨** — `SqlPushdownAdapter.hasCycle(ir)`(DFS 기반). 루프 도입 시 “허용된 루프” 예외 처리만 추가하면 됨(상단 「이미 구현된 항목」 참고). |

### 3-3. T_RUN_JOB 확장 / 롤백·재시도

| 항목                    | 내용                                                                                                                                                                                                                                                                                                                    |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **구현 가능성**         | ✅ **가능**. 서브잡 호출은 `ExecutionService.execute(subJobId)` 재귀 호출로 구현. 롤백/재시도는 트랜잭션 경계와 재시도 정책 설정으로 구현.                                                                                                                                                                              |
| **현재 로직 적용·이식** | ✅ **이식 가능**. `T_RUN_JOB`는 이미 `ComponentType`에 있음. `SqlPushdownAdapter.executeNode`에서 해당 타입일 때 `plan.ir.context`에서 서브잡 ID를 읽어 `ExecutionService.execute` 호출하면 됨. 롤백은 “Output 노드가 트랜잭션 내에서 실행”되도록 현재 미구현인 “전체 파이프라인 SQL” 또는 트랜잭션 래핑이 선행돼야 함. |

### 문서 내 반영 수정 사항

- **파일 커넥터(1-1)**
  - “Python Worker”는 **선택·보조**로 두고, **기본 실행 경로는 JVM(Kotlin) 기반 FileAdapter**로 할 것 — Plan_ETL의 “Execution Plane: Java/Kotlin 런타임 + 파일 IO”와 맞춤.
- **프론트 테스트(스택 정리·8장)**
  - **Vitest**를 1순위로 명시(Vite 프로젝트이므로 설정·속도 최적화).
- **Rate limiting(7-2, 스택 정리)**
  - **Resilience4j**를 Spring 생태계 기준으로 1순위, Guava/Bucket4j는 대안으로 명시.
- **테스트(8장)**
  - “dbt 스타일 assertion”은 **패턴만 참고**, 실제 구현은 JUnit/Kotest + Kotlin으로 한다고 명시.
