## 검증 기능 설계 (ETL Platform 기준)

이 문서는 “데이터 품질 검증(Validation)”을 **현재 ETL Platform 아키텍처(React Flow Job Designer + Kotlin 실행엔진 + SQL Pushdown/FAP)** 에 실제로 붙일 수 있는 형태로 재정리한 것이다.

> 이 프로젝트에는 이미 “실행 전 정적 검증(validate)”이 존재한다.  
> 여기서 말하는 “검증 기능”은 그 위에 얹는 **데이터 검증(T_VALIDATE)** 을 중심으로 다룬다.

---

## 1) 현재 이미 구현된 검증(Preflight Validate)

실행 시작 전에 엔진이 `validate(plan)`을 호출하며, 에러가 있으면 실행 자체를 막는다.

### 1.1 위치(코드 기준)
- `backend/src/main/kotlin/com/platform/etl/execution/ExecutionService.kt`
  - 실행 전에 `engine.validate(plan)` 수행 → 에러가 있으면 FAILED로 반환
- `backend/src/main/kotlin/com/platform/etl/execution/SqlPushdownAdapter.kt`
  - Pushdown(validate + execute) 구현
- `backend/src/main/kotlin/com/platform/etl/execution/FetchAndProcessExecutor.kt`
  - FAP(validate + execute) 구현

### 1.2 현재 하는 검증(핵심)
- **필수 설정값 누락**
  - Input/Output 노드의 `connectionId`, Output의 `tableName` 등
- **Cycle 여부**
  - Job에 순환 참조가 있으면 실행 불가
- **tMap 타겟 컬럼 존재 여부(부분)**
  - Output 노드에 `config.columns`가 캐시된 경우에만 검증 가능
- **미치환 context 변수**
  - 노드 config 문자열 내부의 `context.xxx` 참조가 plan.context에 없으면 에러

이 레이어는 “설정 미비/명백한 오류”를 빠르게 차단하는 용도이며, 데이터 자체의 품질(값/범위/중복 등)을 검사하지는 않는다.

---

## 2) 우리가 추가하려는 검증: 데이터 검증(T_VALIDATE)

### 2.1 목표

`T_VALIDATE` 노드를 파이프라인 중간에 배치하면:

- 입력 데이터(또는 특정 변환 이후 데이터)에 대해 **룰 기반 검증 SQL을 자동 생성/실행**하고
- 결과를 **Execution Logs/Execution Result**에 남기며
- 정책에 따라:
  - **FAIL**: 잡 전체 실패로 종료(또는 Trigger ON_ERROR 경로로 흐르게)
  - **WARN**: 로그만 남기고 계속 진행
  - **REJECT**: (장기) REJECT 라인으로 위반 Row를 별도 스트림 처리

### 2.2 현재 상태(중요)
- `ComponentType.T_VALIDATE`는 enum에 존재한다.
- 하지만 실행 엔진에서 `T_VALIDATE` 실행 로직은 아직 없다(현재는 사실상 SKIPPED 범주로 취급될 가능성이 높음).
- 프론트 팔레트에는 `T_VALIDATE`가 있지만, 속성 패널 UI는 별도 구현이 필요하다.

---

## 3) 구현 전략: 현재 엔진 구조에 맞춘 단계별 구현

현재 실행 엔진은 2가지 경로로 나뉜다.

- **SQL Pushdown 경로**: CTE 기반 컴파일 후 Output에서 `INSERT ... SELECT`
- **Fetch-and-Process(FAP) 경로**: 소스 스트리밍 + 인메모리 변환 + 배치 write

검증도 이 2경로에 맞춰 “작게 시작 → 확장” 방식으로 가는 것이 안전하다.

---

## 4) Phase별 구현 로드맵 (추천)

### Phase A (최소 구현, 추천): Dataset Validation (COUNT 기반) + 실패 정책

**아이디어**
- 검증은 Row를 전체 조회하지 않고 DB에서 `COUNT(*)` 등 집계로 판정한다.
- 위반 Row는 저장/표시하지 않고, **룰별 실패 건수만** 남긴다.

#### A-1) SQL Pushdown 경로
- `SqlPushdownCompiler`에 “검증용 SQL 생성” 함수를 추가한다.
- 검증 대상 데이터셋을 서브쿼리로 감싸고 룰별로 COUNT SQL 생성:
  - `SELECT COUNT(*) FROM (<datasetSql>) v WHERE NOT(<predicate>)`
- `T_VALIDATE` 실행 시점은 2가지 중 하나로 선택 가능:
  - **옵션 1(직관적)**: 실행 루프에서 `T_VALIDATE` 노드에 도달했을 때 해당 지점의 검증 SQL을 실행
  - **옵션 2(구현이 단순할 수 있음)**: Output 실행 전에 upstream의 `T_VALIDATE`들을 모아서 “검증 먼저” 일괄 실행

#### A-2) FAP 경로
FAP는 이미 소스 쿼리(srcQuery)를 구성해 실행한다. Phase A에서는 다음이 가장 단순하다.
- 소스 DB에서 COUNT만 수행:
  - `SELECT COUNT(*) FROM (<srcQuery>) v WHERE NOT(<predicate>)`
- 이 방식은 “변환 이후” 기준 검증은 못하지만, 최소 검증 기능은 빠르게 제공 가능하다.

#### A-3) 결과 반영 방식(권장)
- 각 룰마다 로그를 남긴다.
  - 예: `[VALIDATE] <nodeLabel> rule=<ruleId> failed=<cnt> policy=<FAIL|WARN>`
- policy가 FAIL이고 failed>0이면:
  - `T_VALIDATE` 노드 결과를 FAILED 처리(Trigger 모드면 ON_ERROR로 흐르게 가능)
  - Trigger 모드가 아니면 overall FAILED로 즉시 종료

**장점**
- 구현 난이도/리스크 낮음
- 성능 좋고 DB 친화적

**제약**
- “어떤 행이 왜 실패했는지” 상세 Row 수준은 제공하지 않음

---

### Phase B (중간): 샘플 Row 캡처(옵션) + 룰 결과 UI

**Pushdown**
- 룰 위반 Row를 일부 샘플링:
  - `SELECT * FROM (<datasetSql>) v WHERE NOT(<predicate>) LIMIT 100`
- 샘플을 `PreviewGrid`로 표시 가능(Execution Result에 샘플 포함 or 별도 API)

**FAP**
- 스트리밍 처리 중 룰을 인메모리로 평가하여:
  - rejectCount 집계
  - rejectSamples(최대 N행)만 저장

> 주의: FAP에서 “범용 expression 평가 엔진”을 만들면 복잡도가 커진다.  
> Phase B는 **아주 제한된 룰 타입**(NOT_NULL/RANGE 등)부터 확장하는 것이 현실적이다.

---

### Phase C (장기): REJECT 라인 시맨틱 완성

IR에 `LinkType.REJECT`가 존재하지만 실행 엔진은 미구현(로드맵에도 존재).

- `T_VALIDATE`가 위반 Row를 `REJECT`로 흘리고,
- 별도 Sink(`T_LOG_ROW` 확장 또는 전용 노드)가 REJECT를 저장/출력하는 Talend식 흐름 완성

## 5) T_VALIDATE 노드 config 스펙(권장, 최소 JSON)

현재 노드 config는 `Map<String, Any?>` 기반이므로, 최소 스펙은 아래처럼 시작하는 것을 추천한다.

```json
{
  "connectionId": "UUID (선택)",
  "scope": "UPSTREAM | NODE",
  "policy": "FAIL | WARN",
  "rules": [
    {
      "id": "not_null_customer_id",
      "name": "customer_id NOT NULL",
      "type": "SQL_PREDICATE",
      "predicate": "customer_id IS NOT NULL",
      "message": "customer_id는 NULL일 수 없습니다"
    }
  ]
}
```

- **connectionId**
  - Pushdown: 대개 Output DB에서 검증 SQL을 실행하므로 생략 가능(컴파일 결과 connectionId 사용)
  - FAP Phase A: 소스 DB에서 COUNT를 실행해야 하므로 Input connectionId를 기본으로 사용(명시 override 허용)
- **scope**
  - `UPSTREAM`: 이 Validate 노드의 upstream 데이터셋에 대해 검증
  - `NODE`: 특정 노드 결과(예: 특정 tMap 이후) 기준 검증  
    - Pushdown에서는 “해당 노드까지의 CTE/서브쿼리”로 구성 가능
    - FAP에서는 Phase A에서는 제한(보통 srcQuery 기준), Phase B에서 확장
- **policy**
  - `FAIL`: 위반 건수 > 0이면 노드 FAILED 처리
  - `WARN`: 로그만 남기고 SUCCESS 처리
- **rules**
  - 최소 구현은 `SQL_PREDICATE`(사용자가 WHERE predicate를 입력) 1종으로 시작
  - 이후 UI 친화적 룰(NOT_NULL/RANGE/REGEX/UNIQUE 등)을 추가하고 내부적으로 predicate로 변환

## 6) UI/UX 제안 (현재 Job Designer 기준)

Talend처럼 거대한 DQ 프레임워크를 붙이기보다 “빠르게 쓰는 최소 폼”이 적합하다.

- **검증 정책**: FAIL / WARN 토글
- **검증 대상**
  - upstream 기준(기본)
  - (다중 Output이면) Output 선택/연결 기반으로 대상 결정
- **룰 목록**
  - 룰 추가/삭제
  - 필드: id(자동 생성 가능), name, predicate, message
- **결과 표시**
  - Execution Logs에 룰별 failed count 출력
  - (Phase B) failed 샘플을 PreviewGrid로 확인

## 7) 기존 문서 대비 변경/보완한 점

기존 문서는 Great Expectations/Soda/dbt 등 “외부 DQ 프레임워크” 중심이라 방향성은 좋지만,
현재 프로젝트에 바로 적용하기엔 아래가 빠져 있었다.

- **정적 검증(validate) vs 데이터 검증(T_VALIDATE) 층위 구분**
- **SQL Pushdown vs FAP 2경로에 따른 구현 차이**
- **REJECT 라인(장기)과 COUNT 검증(단기)의 단계 분리**

그래서 이 문서는 “현 코드에 붙일 수 있는 구현 단계(Phase A/B/C)”로 재구성했다.

---

## 8) 결론(추천 순서)

- **1순위**: Phase A로 `T_VALIDATE`를 “COUNT 기반 Dataset Validation”으로 구현
- **2순위**: Phase B로 샘플 Row 캡처 + 결과 UI 강화
- **3순위**: Phase C로 REJECT 라인 시맨틱 완성
