## tMap Expression Builder 설계 메모

Talend tMap Expression Builder의 전체 기능과 구조를 정리하고, 현 ETL Platform에서 어떤 식으로 축약/적용할지에 대한 기준 문서.

---

## 1. Talend tMap Expression Builder 개요

- **역할**
  - tMap 내부에서 각 **Output 컬럼**/**Var(변수)** 에 대해 **Java 기반 표현식**을 작성·검증·관리하는 전용 편집기.
  - GUI 매핑(라인 연결)로는 표현하기 어려운 변환 로직을 코드로 정의하는 레이어.
- **연결 위치**
  - tMap 메인 에디터 상단: Input/Output/Var 테이블 + 매핑 라인.
  - tMap 하단: `Schema Editor` / `Expression Editor` 탭.
  - 개별 셀 옆의 `...` 버튼으로 띄우는 팝업: **Expression Builder 다이얼로그**.

> 핵심: Map Editor가 “구조/라인/컬럼 단위 설정”이라면, Expression Builder는 그 각 컬럼에 매달린 **코드 조각 편집기**.

---

## 2. Expression Builder 화면 구조 (Talend 기준)

### 2.1 팝업 다이얼로그 레이아웃

Talend 공식 문서를 기준으로 한 일반적인 Expression Builder 다이얼로그 UI:

- **좌측 패널 – 리소스 트리**
  - **Input 컬럼 트리**
    - `row1`, `row2` 등 입력 플로우별로 컬럼 리스트 노출.
    - 더블클릭/드래그 시 우측 코드에 `${row1.col}` 형태의 참조 삽입.
  - **Var(변수) 트리**
    - tMap Var 테이블에 정의된 중간 변수들.
    - 복잡한 표현식을 Var로 분리해 재사용.
  - **Global/Context/기타 리소스**
    - `context`, `globalMap`, 루틴(routines) 등 Talend 전역 객체 및 유틸.

- **우측 상단 – 코드 에디터**
  - 현재 선택된 **Output 컬럼 또는 Var**에 대한 단일 표현식을 작성하는 텍스트 영역.
  - 특징:
    - Java 구문 하이라이팅.
    - 괄호, 따옴표 등 기본적인 편집 보조.
    - 다중 줄 코드도 허용하지만, 최종적으로는 “한 컬럼의 리턴값”으로 간주.

- **우측 하단 – 함수/루틴 라이브러리**
  - 카테고리별 함수 목록:
    - `StringHandling` (UPCASE, TRIM, EREPLACE 등)
    - `TalendDate` (parseDate, addDate 등)
    - 수치/논리/집계 관련 유틸
  - 함수 선택 시:
    - 시그니처, 파라미터 설명, 간단한 예제를 하단에 표시.
    - 코드 에디터에 템플릿 삽입: `StringHandling.UPCASE( row1.name )` 형태.

- **하단 버튼 영역**
  - **Check / Test**: 구문 검사 또는 샘플 값 기반 평가.
  - **OK / Cancel**: 표현식 적용/취소.

### 2.2 Map Editor 하단 Expression Editor 탭

팝업 Builder 외에, tMap 메인 UI 하단에는 **Expression Editor** 탭이 존재:

- **전체 tMap 표현식 뷰**
  - 모든 Output 컬럼/Var에 대한 표현식을 Java 코드 형태로 한 번에 보여줌.
  - 개발자는 여기서 “대량 편집(Find & Replace)”를 할 수 있음.
- **동기화**
  - 상단 그리드의 각 셀 Expression과 하단 코드 뷰가 양방향 동기화.
  - 하단에서 수정 후 저장하면, 각 컬럼의 Expression 문자열에 반영.

## 3. Expression Builder 기능 목록 (Talend 기준 “전체 기능”)

### 3.1 개별 표현식 편집

- Output 컬럼/Var 선택 → `...` 버튼 → Expression Builder 팝업.
- 해당 대상에 연결된 **단일 Expression 문자열**만 로딩·수정.
- OK 시:
  - tMap 설정의 해당 컬럼 Expression 필드에 문자열로 저장.
  - Map Editor 상단 그리드에도 즉시 반영.

### 3.2 다중 컬럼/배치 편집

- Map Editor에서 복수 Output 컬럼 선택 후:
  - 공통 Prefix / Suffix / Wrapper를 한 번에 적용하는 기능.
  - 예: 모든 숫자 컬럼에 `TalendString.replaceNull(row.col, 0)` 패턴 일괄 적용.
- 내부적으로는 선택된 컬럼들 각각의 Expression 문자열을 동일 규칙으로 변환.

### 3.3 함수/루틴 라이브러리 탐색

- 카테고리/이름 기반 검색:
  - 문자열, 날짜, 수치, 논리, null 처리 등으로 그룹화.
  - 이름 검색/필터.
- 함수 상세:
  - 시그니처: `StringHandling.UPCASE(String str)`
  - 설명: “문자열을 대문자로 변환”.
  - 예제: `StringHandling.UPCASE(row1.NAME)`.
- 코드 삽입:
  - 함수 더블클릭 시 현재 커서 위치에 템플릿 삽입.

### 3.4 리소스(컬럼/컨텍스트/글로벌) 삽입

- Input/Var/Context/GlobalMap 등 트리에서:
  - 더블클릭/드래그 → 표현식 내에 참조 코드 삽입.
  - 예: `row1.customer_id`, `context.ENV`, `(String)globalMap.get("KEY")`.

### 3.5 표현식 테스트/검증

- **구문 체크**
  - Java 컴파일러 레벨까지는 아니더라도, 기본적인 구문 오류(괄호, 타입 등)를 사전 검출.
- **샘플 데이터 평가**
  - 일부 버전에서는 샘플 입력값을 넣어 Evaluate를 실행해 결과 미리보기.
  - 특히 복잡한 날짜 처리, regex, null 처리 로직 검증에 사용.

### 3.6 검색/치환 & 네비게이션

- Expression Editor(하단 탭) 기준:
  - 전체 tMap 표현식에 대한 **Find / Replace**.
  - 특정 컬럼 이름이나 함수 패턴을 빠르게 찾아서 한 번에 수정.

## 4. Talend tMap Expression Builder의 설계 포인트

1. **Map 구조와 코드 레이어의 분리**
   - 노드/엣지/컬럼 구조는 시각적으로 정의.
   - 실제 변환 로직은 Builder에서 표현식으로 정의.
2. **강한 Java 의존성**
   - 결국 Talend Job은 Java 코드로 생성되므로, Expression도 Java 구문이 1급 시민.
   - Talend 내장 유틸(`StringHandling`, `TalendDate`, 루틴 등)을 중심으로 DSL화.
3. **점진적 복잡도**
   - 단순 매핑(라인 연결) → 컬럼 단위 Expression → Var로 추상화 → Expression Editor 전체 코드 편집.
4. **DX(Developer Experience) 우선**
   - 트리/함수 팔레트/자동 삽입/하이라이팅/테스트 등으로 “IDE 같은 경험” 제공.

## 5. ETL Platform용 Expression Builder로 번역 (간소화 설계)

이 프로젝트는 Java 코드 생성이 아니라 **SQL Pushdown + 인메모리 엔진**이 핵심이므로, Talend 수준의 Java Builder를 그대로 복제하면 오히려 과하다. 따라서 다음 원칙으로 축약:

### 5.1 Builder를 어디에 두는가?

- 별도 팝업보다는 **현재의 `MappingEditorModal`이 곧 Expression Builder** 역할을 수행.
  - 좌측: Input 컬럼/소스 노드.
  - 우측: Output 컬럼 + Expression + Type.
  - 상단/하단의 버튼/배너가 Talend Builder 하단 기능(Test/Enhancements)을 대신.

### 5.2 기능 매핑표 (Talend → ETL Platform)

- **개별 표현식 편집**
  - Talend: Output 셀 `...` → 팝업.
  - 여기: 각 행의 `Expression` 인풋 박스가 바로 Builder의 코드 영역.

- **함수/루틴 라이브러리**
  - Talend: 왼쪽/하단에 방대한 함수 리스트.
  - 여기: 
    - 상단 `Enhancements` 드롭다운(이미 존재)을 “자주 쓰는 패턴 모음”으로 사용.
    - 추후, 우측에 작은 **함수 팔레트 패널** 추가(문자열/날짜/숫자/null 등 10~20개).

- **리소스 트리(컬럼/컨텍스트)**
  - Talend: 좌측 트리에서 컬럼/Context/GlobalMap 삽입.
  - 여기:
    - Input 컬럼 트리는 이미 구현되어 있고, 클릭 → 행 선택 구조로 매핑 설정.
    - Context 변수(`JobIR.context`)는 Expression 안에서 `${BIZ_DT}` 같은 형태로 사용(별도 팔레트는 `ContextVarsPanel`이 담당).

- **표현식 테스트/검증**
  - Talend: Evaluate + 샘플 값 테스트.
  - 여기:
    - Tier 1: 현재 구현된 **타입 호환 경고(⚠️/❌ + 배경색)**.
    - Tier 2: Enhancements(일괄 Expression 패턴 적용).
    - Tier 3(로드맵): 샘플 데이터 + Expression 적용 결과를 `PreviewGrid`에 띄우는 구조로 확장.

- **전체 코드 편집(하단 Expression Editor)**
  - Talend: Java 코드 전체 편집.
  - 여기:
    - SQL/IR는 백엔드에서 생성하므로, 프론트에서는 **전컬럼 Expression 문자열 일괄 편집 기능만 제한적으로** 고려.
    - 예: `MappingEditorModal` 상단에 “JSON 다운로드/업로드” 수준의 일괄 수정만 허용.

### 5.3 UX 가이드

- **IDE 느낌보다는 “지도 + 간단 코드” 느낌**
  - 컬럼 간 매핑과 타입 정보를 시각적으로 강하게 보여주고,
  - Expression은 짧은 스니펫 위주(Trim/Null-safe/Uppercase/간단 산술)로 유도.
- **위험한 자동 변환 최소화**
  - Talend의 강력한 Java 표현식은 자유도가 높은 대신 런타임 에러 리스크가 큼.
  - 이 프로젝트에서는:
    - Auto Map의 기본 Expression은 빈 값(passthrough) 유지.
    - 타입 변환은 추천/경고 위주(사용자가 수동 선택).

## 6. 정리: 이 문서의 활용 방법

- **tMap 관련 기능 설계·리뷰 시 기준점**
  - “Talend는 여기까지 하지만, 우리는 어디까지 할 것인가?”를 결정할 때 참고.
- **tMap 고도화 문서(`tMap고도화.md`)와의 연결**
  - Tier 0/1/2/3 설계 시, 이 Builder 구조를 레퍼런스로 삼아,
  - 어떤 기능이 Expression Builder 레벨에서 처리되어야 하는지 명확히 구분.
