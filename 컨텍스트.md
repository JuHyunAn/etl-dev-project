# 컨텍스트(Context) 변수 전략

> 최종 검토: 2026-03-11
> 현재 구현 상태를 기준으로 실제 동작과 한계, 상용 ETL 비교, 개선 우선순위를 정리합니다.

---

## 1. 현재 구현 — 실제 동작 방식

### 1.1 문법 및 패턴

```
context.변수명
```

- 정규식: `context\.([A-Za-z_][A-Za-z0-9_]*)`
- 변수명 첫 글자: 영문자 또는 `_`
- 예: `context.BIZ_DT`, `context.BATCH_ID`
- **Talend와 동일한 문법** — `context.XXX` 그대로 사용

### 1.2 저장 위치

| 레벨 | 저장 위치 | 형태 |
|------|-----------|------|
| **Job 정의** | `jobs.ir_json` (JSONB) → `JobIR.context` | `Map<String, String>` |
| **스케줄 Step 오버라이드** | `schedule_steps.context_overrides` (JSONB) | JSON 문자열 |

### 1.3 실행 시 Context 머징 규칙

```
최종 context = ir.context (Job 정의값) + runtimeContext (실행 시 전달값)
              → 같은 키는 runtimeContext 값이 우선
```

- **수동 실행**: UI에서 정의한 context 변수가 `executionApi.run(jobId, ctxMap)`으로 전달
- **스케줄 실행**: `step.contextOverrides` JSON이 runtimeContext로 사용

### 1.4 치환 타이밍

```
실행 시작
  → validate(): 미치환 변수 존재 여부 검사
  → resolvedPlan: IR 전체 노드 config를 한 번에 치환 (실행 전)
  → SqlPushdownCompiler: 이미 치환된 값으로 SQL 생성
```

- 치환은 **실행 전 1회 일괄 처리** (SQL Pushdown 방식과 정합)
- 미치환 변수가 있으면 **실행 자체를 막고 오류 반환**

### 1.5 현재 지원 기능 요약

| 기능 | 지원 여부 |
|------|----------|
| Job 레벨 context 변수 정의/저장 | ✅ |
| 실행 시 runtime context 전달 | ✅ |
| 스케줄 step별 contextOverrides | ✅ (JSON textarea) |
| Job context + runtime 머징 (runtime 우선) | ✅ |
| 미치환 변수 실행 전 검증 | ✅ |
| **기본값(default value) 설정** | ❌ |
| **변수 타입 지정 (Date/Int/String)** | ❌ (전부 String) |
| **Context 그룹 (dev/prod 환경 분리)** | ❌ |
| **내장 함수 (`${today()}`, `${uuid()}`)** | ❌ (계획만 있음) |
| **스케줄 context 오버라이드 UI (개별 변수)** | ❌ (raw JSON만) |
| **재귀/중첩 치환** | ❌ |

## 2. 상용 ETL(Talend) 대비 격차 분석

### 2.1 Talend이 제공하는 Context 기능

| 기능 | Talend | 현재 플랫폼 |
|------|--------|------------|
| 문법 | `context.VAR` | ✅ 동일 |
| 변수 타입 | String, Integer, Date, Boolean, Object | ❌ String만 |
| **기본값(Default Value)** | 각 변수마다 설정 가능 — 값 없이 실행해도 동작 | ❌ 없으면 실행 실패 |
| **Context 그룹** | Built-In / Default / Production / Development 분리 | ❌ 없음 |
| **Context 프롬프트** | 실행 전 대화상자에서 값 입력 | ❌ 없음 |
| **내장 시스템 변수** | `TalendDate.getDate()`, `TalendString.getUUID4()` 등 | ❌ 없음 |
| **Context Import/Export** | .properties 파일로 환경별 값 관리 | ❌ 없음 |
| **스케줄 오버라이드 UI** | 각 변수별 입력 폼 제공 | ❌ raw JSON |

### 2.2 현재 가장 큰 활용성 저하 원인

#### ① 기본값 없음 → 실행 실패 위험
- Talend: `context.BIZ_DT`의 기본값을 `today()`로 설정하면, 값을 주지 않아도 오늘 날짜로 실행
- 현재: 값 없으면 validate()에서 오류 → 실행 불가
- **영향**: 스케줄에서 contextOverrides를 빠뜨리면 Job이 실행 안 됨

#### ② 타입 없음 → 날짜/숫자 처리 SQL에서 직접 처리해야 함
- Talend: `Date` 타입 변수는 자동으로 포맷 변환
- 현재: `context.BIZ_DT = "20260311"` → SQL에서 `TO_DATE(context.BIZ_DT, 'YYYYMMDD')` 직접 작성 필요

#### ③ 스케줄 contextOverrides가 raw JSON
- 실제로 변수명을 알고 JSON을 직접 입력해야 하므로 **실수 유발 가능성 높음**
- Job에 정의된 context 변수 목록을 스케줄 UI에서 보여주지 않음

## 3. Run-단위 vs Row-단위 변수 분류 (변경 없음)

### 3.1 Run-단위 변수 (현재 지원 범위)
- 실행 1회 동안 값이 고정
- 예: `BIZ_DT`, `BATCH_ID`, `RUN_UUID`
- 현재 문자열 치환 방식과 완전히 맞음

### 3.2 Row-단위 생성 로직 (현재 미지원, DB 함수로 대체)
- 행마다 다른 값 필요 (순번, 유니크 키 등)
- SQL Pushdown 구조상 "실행 전 치환"만으로 불가능
- **권장**: SQL 표현식에서 DB 기능으로 직접 해결
  - 순번: `ROW_NUMBER() OVER (ORDER BY ...)`
  - 유니크: `gen_random_uuid()` (PostgreSQL)
  - 시퀀스: `nextval('seq_name')`

## 4. 개선 우선순위 (활용성 관점)

### Priority 1 — 기본값(Default Value) 지원 ★★★

**문제**: 값 없으면 실행 실패 → 스케줄 안정성 저하
**해결**: Context 변수에 `defaultValue` 필드 추가

```
변경 전: Map<String, String>
변경 후: Map<String, ContextVar(value, defaultValue, description)>
```

- validate() 수정: 미치환 변수가 있어도 defaultValue가 있으면 허용
- 치환 순서: runtimeContext → ir.context value → ir.context defaultValue

### Priority 2 — 스케줄 contextOverrides UI 개선 ★★☆

**문제**: raw JSON 입력 → 변수명 오타, 누락 위험
**해결**: Job에 정의된 context 변수 목록을 스케줄 Step UI에서 자동 노출

- Schedule 설정 화면에서 해당 Job의 context 변수 목록을 키별 입력 폼으로 표시
- JSON 생성은 백엔드가 처리

### Priority 3 — 내장 Run-단위 함수 ★★☆

**문제**: `오늘 날짜`, `실행 UUID` 등을 항상 수동으로 contextOverrides에 지정해야 함
**해결**: context value에 특수 문법 허용, 실행 시작 시 서버가 평가

```
context.BIZ_DT = ${today('yyyyMMdd')}   → 실행 시 "20260311"로 치환
context.RUN_ID  = ${uuid()}              → 실행마다 새 UUID
context.BIZ_DT_PREV = ${dateAdd(context.BIZ_DT, -1)}
```

화이트리스트 함수만 허용 (임의 코드 실행 금지):
- `today(format)` / `now(format)` — 날짜/시간
- `uuid()` — UUID v4
- `dateAdd(date, days)` — 날짜 계산
- `env(KEY)` — 서버 환경변수 (허가된 키만)

### Priority 4 — 변수 타입 및 Context 그룹 ★☆☆

Talend 수준의 타입 시스템(Date, Integer 등)과 환경 그룹(dev/prod)은
현재 단계에서 복잡도 대비 효용이 낮으므로 후순위 고려.

## 5. 반복(Loop) 요구 처리 원칙 (변경 없음)

| 반복 종류 | 권장 방식 |
|-----------|----------|
| 기간 반복(일별/월별) | Quartz 스케줄러가 Job을 반복 실행 + context.BIZ_DT 주입 |
| 테이블/파티션 반복 | T_RUN_JOB으로 서브잡 반복 호출 |
| 행 단위 반복 | DB 윈도우 함수/시퀀스 (ROW_NUMBER, nextval) |

## 6. 외부 피드백 분석 (2026-03-11)

> 피드백 원문 요지: 컨텍스트 패널 UX에서 직접값/함수값 구분, 함수 선택 팝오버, tMap 함수 팔레트 + 스니펫 삽입 UI, Row-단위는 "DB 함수 + tMap Function Palette"로 처리

### 6.1 차용할 부분 ✅

#### ① 직접값(Literal) vs 함수값(Function) 구분 UI
- Context 변수 입력 시 두 가지 모드로 분리:
  - `Literal`: 고정 문자열 직접 입력 (예: `20260101`, `BATCH_001`)
  - `Function`: 내장 함수 선택 팝오버 (예: `today('yyyyMMdd')`, `uuid()`)
- **Priority 3 내장 함수 구현 시 이 UX 패턴 채택** — 텍스트박스에 `${...}` 수동 입력 대신 모드 전환 UI

#### ② 함수 선택 팝오버 + 폼 입력 + 예시 프리뷰
- 함수 카테고리(날짜/ID/환경) → 함수 선택 → 파라미터 폼 → 실시간 프리뷰 ("→ 20260311")
- **ContextVarsPanel 구현 시 적용**: 단순 `<input type="text">` 대신 이 팝오버 플로우 사용

#### ③ tMap 표현식 에디터에 함수 팔레트 + 스니펫 삽입
- tMap 컬럼 Expression 입력 시 "함수를 고른다" 수준에서 작업 가능하도록
- 현재 tMap Auto Map에서 `TRIM(col)`, `COALESCE(col, 0)` 자동 적용 중 → 수동 편집 시에도 함수 스니펫 삽입 UI 제공
- **사용자가 직접 Java/SQL 코드를 작성하는 방향 금지** → 팔레트 선택 → 생성된 표현식 삽입

#### ④ Row-단위 = DB 함수 + tMap Function Palette (기존 원칙 보강)
- 기존 컨텍스트.md의 "DB 윈도우 함수/시퀀스" 방향과 일치
- 추가: tMap Expression에서 DB 함수 스니펫(`ROW_NUMBER() OVER (...)`, `gen_random_uuid()`)을 팔레트로 제공하면 사용자가 SQL 문법 없이도 Row-단위 처리 가능

---

### 6.2 보완할 부분 ⚠️

| 피드백 내용 | 보완 방향 |
|------------|----------|
| 함수 팔레트는 tMap만 언급 | Context 변수 기본값 설정 시에도 동일 팔레트 재사용 (UI 컴포넌트 공유) |
| 함수 카테고리/툴팁/예제 정리 필요 | Priority 3 구현 전 함수 명세를 별도 테이블로 작성 후 진행 |
| 스니펫 삽입 UI 구현 방식 미언급 | React 스택: 팝오버는 Tailwind + `useState`, 스니펫 삽입은 커서 위치 기반 `textarea` 조작 |

---

### 6.3 현재 단계에서 불필요/후순위 ❌

| 피드백 내용 | 판단 이유 |
|------------|---------|
| JSQLParser (표현식 파서) | 현재 SQL Pushdown 구조는 컴파일된 SQL을 그대로 전달 — 런타임 SQL 파싱 불필요. 복잡도 대비 효용 낮음 |
| "직접 Java 코드 작성 금지" 방향 | 우리 플랫폼은 이미 Java 코드 직접 작성 방식이 아님 — 별도 조치 불필요 |
| 함수 매핑 (코드-to-DB 함수 매핑) | 현재 String 치환 방식으로 충분. 함수 평가는 서버측 화이트리스트 방식으로 이미 설계됨 |

---

### 6.4 구현 순서에 미치는 영향

```
Priority 1 (기본값) → 2 (스케줄 UI) 완료 후
Priority 3 (내장 함수) 구현 시:
  1) 함수 명세 테이블 먼저 작성 (today/uuid/dateAdd/env)
  2) 백엔드: SqlPushdownAdapter 함수 평가 로직
  3) 프론트: ContextVarsPanel에 직접값/함수값 모드 전환 UI
  4) 프론트: tMap Expression에 동일 팔레트 재사용
```
