# 스타 스키마 설계 및 데이터 웨어하우스(DW) 구축 실무 가이드 (AGENT 교육용)

본 문서는 BI 도구(Power BI 등) 활용을 위한 데이터 웨어하우스 설계의 표준 지침서입니다. AGENT는 본 가이드를 바탕으로 ODS, DW, DM 단계별 테이블을 설계하고 속성을 정의해야 합니다.

---

## 1. 차원 모델링의 핵심 프로세스 (Kimball 4-Step)

AGENT는 모든 모델링 작업 시 다음 4단계를 순차적으로 수행하며 각 단계의 의사결정 근거를 기록해야 한다.

### Step 1: 비즈니스 프로세스 선택 (Select the Business Process)
* **정의:** 분석 대상이 되는 비즈니스 활동(이벤트)을 식별한다.
* **AGENT 수칙:** "무엇을 보고 싶은가?"가 아닌 "어떤 시스템 이벤트가 데이터를 생성하는가?"를 기준으로 정의한다. (예: 판매, 재고 이동, 웹 로그 클릭)

### Step 2: 그레인(Grain) 선언
* **정의:** 사실 테이블의 행(Row) 하나가 물리적으로 무엇을 의미하는지 정의한다.
* **AGENT 수칙:** * 가능한 한 가장 상세한 수준(Atomic Grain)을 선택한다. 
    * "일자별-상점별-상품별 판매 내역"과 같이 구체적으로 명시한다.
    * 그레인이 혼재된 사실 테이블을 설계해서는 안 된다.

### Step 3: 차원 식별 (Identify the Dimensions)
* **정의:** 사실 테이블의 측정값에 컨텍스트(Context)를 제공하는 "누가, 언제, 어디서, 무엇을"을 정의한다.
* **AGENT 수칙:** 그레인이 결정되면 차원은 자연스럽게 도출되어야 한다. (예: 날짜, 상품, 매장, 고객, 프로모션 등)

### Step 4: 사실 식별 (Identify the Facts)
* **정의:** 비즈니스 프로세스에서 발생하는 수치적 측정값을 식별한다.
* **AGENT 수칙:** * 사실은 반드시 수치 데이터(Numeric)여야 한다.
    * 계산된 필드보다는 원천 데이터를 우선 적재한다.

---

## 2. 데이터 계층별 설계 표준 (ODS → DW → DM)

### 2.1 ODS (Operational Data Store)
* **목적:** 원천 시스템 데이터의 복제 및 통합 전초 기지.
* **설계 규칙:**
    * **1:1 복제:** 원천 시스템의 테이블 구조와 컬럼명을 최대한 유지한다.
    * **메타데이터 추가:** `ETL_LOAD_DATE`, `SOURCE_SYSTEM_ID` 등 관리용 컬럼을 반드시 추가한다.
    * **비즈니스 로직 배제:** 데이터 변환을 최소화하고 정규화된(3NF) 상태를 유지할 수 있다.

### 2.2 DW (Data Warehouse) - Integration Layer
* **목적:** 여러 ODS의 데이터를 전사 관점에서 통합하고 정제.
* **설계 규칙:**
    * **데이터 정제:** Null 값 처리, 데이터 형식 통일(예: Y/N -> 1/0).
    * **통합 차원 생성:** 여러 시스템에 흩어진 고객/상품 정보를 하나로 통합한다.
    * **이력 관리:** SCD(Slowly Changing Dimension) 로직을 적용하여 과거 데이터를 보존한다.

### 2.3 DM (Data Mart) - Star Schema Layer
* **목적:** BI 툴에서 직접 참조하는 분석용 최적화 레이어.
* **설계 규칙:**
    * **스타 스키마 구조:** 중앙의 Fact Table과 이를 둘러싼 Dimension Table로 구성한다.
    * **비정규화:** 조인 횟수를 줄이기 위해 차원 테이블은 완전히 비정규화(Flat Table)한다.

---

## 3. 상세 테이블 설계 규칙

### 3.1 사실 테이블 (Fact Table)
1.  **기본키(PK):** 여러 차원의 대리키(SK)를 조합한 복합키 또는 별도의 시퀀스 ID를 사용한다.
2.  **외래키(FK):** 반드시 차원 테이블의 대리키(Surrogate Key)를 참조해야 한다.
3.  **측정값 유형 분류:**
    * **가산적(Additive):** 모든 차원에서 합산 가능 (예: 매출액, 수량).
    * **반가산적(Semi-additive):** 특정 차원(시간)에서는 합산이 무의미 (예: 재고 잔액, 통장 잔고).
    * **비가산적(Non-additive):** 합산이 불가능하여 평균이나 비율로 처리 (예: 단위 가격, 할인율).
4.  **Null 처리:** FK 컬럼에는 Null을 허용하지 않는다. 데이터가 없는 경우 차원 테이블의 '미지정(-1)' 항목을 가리키도록 한다.

### 3.2 차원 테이블 (Dimension Table)
1.  **대리키(Surrogate Key) 필수:** 운영 시스템의 PK 대신 DW 전용 정수형(Int/BigInt) 키를 생성한다. (시스템 변경 대비 및 성능 향상)
2.  **설명적 속성:** 코드(Code)보다는 이름(Name), 구분(Type), 그룹(Group) 등 텍스트 위주의 상세 정보를 포함한다.
3.  **날짜 차원(Date Dimension):** 시스템 날짜 함수 대신 별도의 날짜 차원 테이블을 생성하여 회계연도, 분기, 휴일 여부 등을 관리한다.
4.  **계층 구조(Hierarchy):** '연도 > 분기 > 월 > 일' 혹은 '대분류 > 중분류 > 소분류'와 같은 계층을 단일 테이블 내에 컬럼으로 나열한다.

---

## 4. 이력 관리 기법 (SCD: Slowly Changing Dimension)

AGENT는 데이터의 변화를 어떻게 기록할지 결정해야 한다.

* **Type 1 (Overwrite):** 과거 값을 덮어쓴다. 과거 이력이 필요 없는 경우 사용한다.
* **Type 2 (Add New Row):** 새로운 행을 추가하고 유효 기간을 명시한다. (추천 방식)
    * 필수 컬럼: `START_DATE`, `END_DATE`, `IS_CURRENT_FLAG`
* **Type 3 (Add New Column):** 이전 값을 별도 컬럼에 저장한다. (최근 2단계 이력만 관리 시 사용)

---

## 5. BI(Power BI) 최적화를 위한 핵심 규칙

1.  **데이터 타입 최적화:** 문자열보다는 숫자형(Integer)을 선호한다. 고정 소수점(Decimal)은 금액 계산 시 오차를 줄인다.
2.  **단방향 관계 유지:** 차원(1)에서 사실(*)로 흐르는 단방향 필터 방향을 기본으로 한다.
3.  **컬럼 가독성:** 비즈니스 사용자가 직관적으로 이해할 수 있는 한글 또는 영문 표준 명칭을 사용한다. (예: `PROD_NM` -> `상품명`)
4.  **퇴화 차원(Degenerate Dimension) 활용:** 주문번호처럼 별도 테이블로 만들기엔 크고 사실 테이블에 직접 두기 적합한 속성은 사실 테이블에 그대로 둔다.
