# ETL Platform 제안 기능 — 구현 가능성 및 현재 로직 적용·이식 분석

> WORK_HISTORY.md 및 제안서(미구현 및 핵심 보완 기능)를 기준으로, **현재 코드베이스**와 대조하여 각 제안의 **구현 가능성**과 **기존 구현 로직에의 적용·이식 가능성**을 정리한 문서입니다.

---

## 분석 기준 요약

| 항목 | 현재 구현 상태 (코드 기준) |
|------|----------------------------|
| **실행 엔진** | `ExecutionEngine` 인터페이스 + `SqlPushdownAdapter` 단일 구현. `ExecutionService`에서 `ir.engineType`으로 엔진 선택, `validate()` → `execute()` 호출. |
| **IR** | `JobIR`(nodes, edges, context), `NodeIR`(config 맵), `ComponentType` enum에 T_FILE_INPUT/OUTPUT, T_RUN_JOB, T_VALIDATE, T_PROFILE, T_LINEAGE 포함. `PortIR`에 이미 `REJECT` 포트 타입 정의. |
| **잡/메타** | `Job` 엔티티: `version`, `status`(DRAFT/PUBLISHED/ARCHIVED), `irJson`(JSONB). `executions` 테이블 존재하나 **실행 결과 저장 로직 없음**(Controller가 결과만 반환). |
| **커넥션** | `Connection` 엔티티(DB 타입별 host/port/database/schema). **풀링/Repository 없음** — 매 요청 `DriverManager.getConnection`. |
| **파일 I/O** | 팔레트·타입·노드 색상에 T_FILE_INPUT/OUTPUT 존재. **속성 패널·런타임 처리 없음.** |

## 1. 데이터 소스 및 타겟 연결 관리 (Connectivity Management)

### 1-1. 파일 커넥터 (T_FILE_INPUT / T_FILE_OUTPUT)

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. 스택(Spring Boot, React)만으로 구현 가능. |
| **현재 로직 적용·이식** | ✅ **적용 가능**. `ComponentType.T_FILE_INPUT`, `T_FILE_OUTPUT` 이미 IR·팔레트·CustomNodes에 정의돼 있음. `PropertiesPanel`에 `JdbcInputConfig`/`JdbcOutputConfig` 패턴 그대로 따라 파일 경로·포맷(CSV/JSON 등)·헤더 옵션 UI 추가 가능. **실행 경로**: `SqlPushdownAdapter`는 JDBC만 처리하므로, 파일 컴포넌트는 (1) 별도 `FileAdapter` 구현 후 `ExecutionService`에서 노드 타입별 엔진 분기하거나, (2) **엔진 타입**을 `SQL_PUSHDOWN`과 `FILE`(또는 Python Worker)으로 나누어 IR에서 혼합 DAG 시 “파일 노드만 Python/별도 런타임”으로 위임하는 방식 필요. |
| **리스크** | 로컬 파일 경로 보안·SaaS 시 서버-클라이언트 파일 공유 방식(업로드/스테이징) 설계 필요. S3/GCS는 Connection 확장(타입+설정) 후 어댑터 추가로 이식 가능. |

### 1-2. 확장 커넥터 (NoSQL, API, 빅데이터)

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. 각 소스별 SDK(예: MongoDB, REST 클라이언트) 의존성 추가로 구현 가능. |
| **현재 로직 적용·이식** | ✅ **이식 가능**. 현재 `ConnectionService`는 JDBC 전제(`buildJdbcUrl`, `DriverManager`). **어댑터 패턴** 도입 시: (1) `Connection`에 `connectionType`(JDBC / FILE / API / MONGO 등) 또는 기존 `dbType` 확장, (2) `SchemaService`처럼 “메타데이터 조회”를 인터페이스로 두고 DB별·API별 구현체 주입, (3) 프론트 Connection 생성 UI에 타입 드롭다운 확장. **이식 시 주의**: 스키마/테이블 개념이 없는 소스(API, NoSQL)는 `listTables`/`getColumns` 의미를 “리소스/컬럼 목록”으로 재해석해야 함. |

### 1-3. 자동 스키마 변경 탐지

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. Quartz 또는 Spring Scheduled + DB 메타데이터 해시/버전 저장으로 구현 가능. |
| **현재 로직 적용·이식** | ⚠️ **부분 이식**. 현재는 “연결 시점” 스키마 조회만 있음. 주기적 폴링·알림은 **신규 모듈**(스케줄러, 스키마 스냅샷 테이블, 알림 채널)로 추가. WebSocket은 `CorsConfig`·프론트와 연동해 실시간 푸시 가능. 기존 `SchemaController`/`SchemaService`는 그대로 두고 “스냅샷 비교” 서비스만 추가하면 됨. |

### 1-4. 연결 풀링 및 모니터링

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. HikariCP는 Spring Data JPA 기본 풀(메타 DB용)로 이미 사용 중일 가능성 높음. **대상 DB(ETL 타겟)**용 풀을 별도로 두려면 `Connection`별 DataSource 빈 또는 풀 매니저 도입 필요. |
| **현재 로직 적용·이식** | ✅ **이식 가능**. 현재 `SchemaService`·`SqlPushdownAdapter`는 `ConnectionService.get` + `buildJdbcUrl` + `DriverManager.getConnection`으로 **매 호출 새 연결**. 여기를 “Connection ID → DataSource(풀)” 캐시로 교체하면 되며, `Connection` CRUD 시 풀 생성/폐기 정책만 정하면 됨. UI에 “사용률/풀 상태” 표시는 실행 통계·메트릭 API가 있은 뒤 연결 가능. |

## 2. 데이터 변환 로직 지원 (Transformation Logic)

### 2-1. Expression 유효성 검사

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. JSQLParser 등 SQL 파서 또는 정규식/DSL 파서로 표현식 검증 가능. |
| **현재 로직 적용·이식** | ✅ **적용 가능**. tMap/필터의 `config`(mappings, condition)는 이미 IR에 저장됨. **백엔드**: `ExecutionService`에 `validateExpression(irJson, nodeId, expression)` 같은 메서드 추가하거나, `ExecutionEngine.validate()` 확장으로 “표현식 문법 검사” 항목 추가. **프론트**: PropertiesPanel/MappingEditorModal에서 API 호출 또는 클라이언트 파서로 실시간 검증 후 Monaco/인라인 에러 표시. |

### 2-2. 고급 변환 확장 (ML 클리닝, T_PROFILE/T_VALIDATE 완성)

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. Kotlin/Java 라이브러리 또는 Python 브릿지로 구현 가능. |
| **현재 로직 적용·이식** | ✅ **이식 가능**. `ComponentType`에 이미 `T_VALIDATE`, `T_PROFILE`, `T_LINEAGE` 존재. 팔레트·노드 렌더링만 추가하고, 실행 경로에서 해당 타입을 `executeNode` 분기로 처리하면 됨. ML 클리닝은 “변환 노드” 하나 더 두거나 T_MAP 확장(플러그인 표현식)으로 붙일 수 있음. |

### 2-3. Pushdown 최적화 (SqlPushdownAdapter 완성)

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. IR → Dialect별 SQL 생성기는 설계상 핵심이므로 구현 가능. |
| **현재 로직 적용·이식** | ✅ **직접 적용 대상**. 현재 `SqlPushdownAdapter`는 노드별로 **단순 SELECT/실행**만 하고, CTE/전체 파이프라인 SQL 컴파일은 미구현. Plan_ETL/Main_Plan의 “IR → SQL AST → Dialect SQL” 레이어를 여기에 넣으면 됨. JOIN/FILTER를 소스 쿼리로 흡수하는 최적화는 이 컴파일러 단계에서 처리 가능. EXPLAIN/튜닝 UI는 실행 결과에 `generatedSql`이 이미 있으므로, 별도 “설계 단계 검증” API로 확장 가능. |

### 2-4. 멀티 아웃풋 포트 (REJECT 라인)

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. React Flow handle 추가 + IR 확장. |
| **현재 로직 적용·이식** | ✅ **이식 가능**. `PortType.REJECT`, `LinkType.REJECT`가 이미 `JobIR.kt`에 정의돼 있음. `NodeResult`에 `rowsRejected` 필드 존재. 부족한 부분: (1) `CustomNodes`에서 출력 포트를 “main + reject”로 두 개 노출, (2) 엣지 연결 시 `linkType: REJECT` 저장, (3) `SqlPushdownAdapter`(또는 변환 노드)에서 실패 행을 reject 포트로 내보내는 시맨틱(실제로는 별도 테이블/파일 쓰기로 구현) 정의. |

## 3. 잡/워크플로우 오케스트레이션 (Job Orchestration)

### 3-1. 스케줄링

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. Quartz 또는 Spring `@Scheduled` + DB에 cron 표현 저장. |
| **현재 로직 적용·이식** | ✅ **이식 가능**. `Job` 엔티티·테이블에 `cron_expression`(또는 `schedule_config`) 컬럼 추가, `ExecutionService.execute`를 Quartz Job에서 호출하도록 하면 됨. 실행 시 `triggered_by`는 이미 `executions` 테이블에 컬럼으로 있으나, **현재 실행 결과를 DB에 저장하지 않음** — 스케줄링 전에 실행 이력 저장을 먼저 구현하는 것이 일관적. |

### 3-2. 고급 워크플로우 (조건 분기, 루프)

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. DAG 확장(조건 엣지, 루프 메타노드)으로 표현 가능. |
| **현재 로직 적용·이식** | ⚠️ **부분 이식**. 현재 IR은 **단순 DAG**(위상 정렬로 순서만 결정). 분기/루프는 (1) IR에 “조건식” 엣지 또는 “서브그래프” 노드 추가, (2) `ExecutionService.topologicalSort`를 “조건 평가 후 다음 노드 선택”으로 확장. 기존 `ExecutionEngine` 인터페이스는 그대로 두고, Plan 단계에서 “실행 경로”를 동적으로 결정하는 레이어를 넣으면 됨. |

### 3-3. T_RUN_JOB 확장 / 롤백·재시도

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. 서브잡 호출은 `ExecutionService.execute(subJobId)` 재귀 호출로 구현. 롤백/재시도는 트랜잭션 경계와 재시도 정책 설정으로 구현. |
| **현재 로직 적용·이식** | ✅ **이식 가능**. `T_RUN_JOB`는 이미 `ComponentType`에 있음. `SqlPushdownAdapter.executeNode`에서 해당 타입일 때 `plan.ir.context`에서 서브잡 ID를 읽어 `ExecutionService.execute` 호출하면 됨. 롤백은 “Output 노드가 트랜잭션 내에서 실행”되도록 현재 미구현인 “전체 파이프라인 SQL” 또는 트랜잭션 래핑이 선행돼야 함. |

## 4. 에러 핸들링 및 로깅 (Error Handling and Logging)

| 항목 | 내용 |
|------|------|
| **구현 가능성** | ✅ **가능**. Reject 행 분리, Logback appender, 메일/Slack 등 표준 방식으로 구현 가능. |
| **현재 로직 적용·이식** | ✅ **이식 가능**. `ExecutionModels.kt`에 `NodeResult.errorMessage`, `rowsRejected` 있음. “에러 처리 전략”은 노드 `config`에 `onError: REJECT|FAIL|LOG` 등 추가하면 됨. 중앙 로깅·알림은 기존 `GlobalExceptionHandler`·실행 로그에 덧붙이는 형태로 이식. 실행 결과를 DB에 저장하면 “실패 시 알림”은 스케줄러/이벤트에서 조회해 트리거 가능. |

**우선 권장 순서 (현재 코드 기준)**  
1. **실행 이력 저장** — `executions` 테이블에 INSERT + 목록 API → 스케줄링·모니터링·알림의 기반.  
2. **파일 커넥터** — IR/팔레트 이미 있으므로 속성 패널 + 실행 분기만 추가.  
3. **스케줄링** — Job에 cron 필드 + Quartz 연동.  
4. **AI 프록시** — 키 보안 및 CORS 대응을 위해 백엔드 경유.  
5. **Pushdown 완성** — IR → SQL 컴파일러로 변환/필터를 DB로 푸시.

이 문서는 제안서의 각 기능이 **구현 가능한지**, 그리고 **현재 구현된 로직에 적용·이식 가능한지**를 코드 기준으로 분석한 결과입니다. 실제 구현 시에는 WORK_HISTORY.md의 미구현 과제와 Plan_ETL.md 로드맵과 함께 참고하시면 됩니다.
