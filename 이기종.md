# 이기종 DB 지원 및 대용량 최적화 설계 방안

작성일: 2026-03-13
현재 아키텍처: SQL Pushdown (단일 DB 한정) + CTE 체인 방식

> **문서 읽기 전 주의**
> 각 섹션에 `[현재 구현]` / `[PHASE 1]` / `[PHASE 2]` / `[PHASE 3]` 태그를 명시한다.
> 태그 없이 읽으면 미래 설계를 현재 동작으로 오해할 수 있다.

---

## 1. 현재 구조의 문제 인식 `[현재 구현]`

### 1-1. FULL SCAN 문제

현재 T_JDBC_INPUT은 조건 없이 전체 테이블을 스캔한다.

```sql
-- 현재 생성되는 CTE
cte_input AS (
  SELECT * FROM source_schema.orders   -- 1억 건 전체 스캔
)
```

**문제가 되는 상황:**
- 매일 실행되는 배치인데 전날 이후 변경분만 필요한 경우
- 수억 건 테이블에서 WHERE 조건이 있음에도 CTE 구조상 전체를 먼저 읽는 경우
- 이기종 DB fetch 시 전체 데이터를 네트워크로 전송

### 1-2. 이기종 DB 처리 불가 `[현재 구현]`

현재 모든 SQL(CTE 체인 + INSERT)이 **Output DB의 단일 JDBC 커넥션 하나에서만** 실행된다.
Input 노드의 connectionId는 컴파일 시 schema prefix 추출에만 쓰이고 실행 시 무시된다.
Input 노드가 다른 DB 서버를 바라볼 경우 런타임 에러 발생.

### 1-3. 이기종 JOIN 불가 `[현재 구현]`

T_JOIN은 CTE 이름만 참조하는 구조이므로, 두 Input이 다른 서버에 있으면
같은 SQL 안에서 JOIN 불가. JOIN 전략 선택 로직 자체가 존재하지 않는다.

## 2. 핵심 설계 원칙

> **"데이터는 최대한 소스에서 줄이고, 이동은 최소화하고, 합산은 최적 전략으로"**

세 가지 관심사를 분리하여 설계한다:

```
[1] 얼마나 읽을 것인가?   → 증분/파티션 전략 (FULL SCAN 방지)
[2] 어디서 처리할 것인가? → 실행 경로 라우팅 (Pushdown vs Fetch)
[3] 어떻게 합칠 것인가?  → 조인 전략 (Broadcast vs Hash vs Pushdown)
```

---

## 3. 증분 처리 전략 (FULL SCAN 해결)

### 3-1. 방식 비교

| 방식 | 원리 | 적합한 상황 | 구현 복잡도 | 목표 Phase |
|------|------|------------|------------|-----------|
| **Timestamp Watermark** | `WHERE updated_at >= 마지막_watermark` | 변경 컬럼이 있는 테이블 | ⭐ | PHASE 1 |
| **Offset(PK) 기반** | `WHERE id > 마지막_처리_id` | 단조증가 PK가 있는 경우 | ⭐ | PHASE 1 |
| **파티션 프루닝** | 날짜 파티션 중 해당 파티션만 스캔 | 파티션 테이블 (DW 환경) | ⭐⭐ | PHASE 2 |
| **CDC (Log 기반)** | DB WAL/Redo 로그에서 변경분 캡처 | 실시간, 변경 컬럼 없는 경우 | ⭐⭐⭐⭐ | 별도 트랙 |

### 3-2. 권장 구현 순서

#### PHASE 1 — Watermark 기반 증분 `[설계 목표]`

Input 노드 설정에 증분 옵션 추가 (현재 JobIR에 없는 필드, 추가 필요):

```
incremental:
  enabled: true
  mode: TIMESTAMP          # TIMESTAMP | OFFSET | FULL
  column: updated_at       # 기준 컬럼
  watermarkVar: last_run   # context 변수에 마지막 watermark 저장
```

실행 시 CTE 생성:

```sql
-- FULL SCAN (현재)
cte_input AS (SELECT * FROM orders)

-- Watermark 증분 (개선 목표)
cte_input AS (
  SELECT * FROM orders
  WHERE updated_at >= '2026-03-12 00:00:00'   -- context.last_run 치환
)
```

**⚠️ 경계값 처리 주의사항**

단순 `>` 사용 시 문제:
```
last_run = '2026-03-12 00:00:00' 저장
→ 해당 시각과 동일한 타임스탬프로 나중에 들어온 행 유실 가능
```

안전한 패턴 두 가지:

```
패턴 A: >= 사용 + 타겟 UPSERT(MERGE)로 중복 허용
  WHERE updated_at >= last_run
  → 재처리되는 경계 행은 UPSERT로 멱등성 보장

패턴 B: > 사용 + watermark를 실행 시작 시각으로 저장
  WHERE updated_at > last_run
  저장값 = 실행 시작 시각 (처리 중 삽입된 행은 다음 배치에서 처리)
  → 유실 없음, 단 실행 중 삽입 행은 다음 회차 처리
```

**⚠️ 타임존 이슈**

소스 DB와 ETL 서버의 타임존이 다른 경우 watermark 비교 오작동 가능.
→ watermark는 항상 UTC 기준으로 저장/비교 권장.

실행 완료 후 watermark 갱신 규칙:
- 성공 시에만 갱신 → 실패 시 동일 범위 재처리 (멱등성 보장)
- 갱신 대상: `etl_watermarks` 메타 테이블 (PHASE 1 도입 예정, 현재 미존재)

#### PHASE 2 — 파티션 프루닝 `[설계 목표]`

Input 노드에 파티션 컬럼과 범위를 지정하면,
컴파일러/Fetcher가 WHERE 절에 파티션 조건을 자동 추가:

```sql
cte_input AS (
  SELECT * FROM orders PARTITION (p_20260313)
  -- 또는
  SELECT * FROM orders WHERE order_date = '2026-03-13'
)
```

#### PHASE 3 (별도 트랙) — CDC (Log 기반)

Debezium 등 CDC 툴과 연동하여 변경 이벤트를 Kafka 토픽으로 수신.
현재 배치 아키텍처와 설계 철학이 다름 → 섣불리 혼합하면 복잡도만 증가.
실시간 파이프라인이 요구될 시점에 별도 트랙으로 분리 개발.

## 4. 실행 경로 라우팅 (하이브리드 핵심) `[PHASE 1 설계 목표]`

> 현재는 SqlPushdownAdapter 단일 엔진만 존재. 아래 구조는 PHASE 1에서 도입 예정.

### 4-1. 라우팅 판단 로직

Job 실행 계획 수립 시 IR을 분석하여 노드 그룹을 자동 분류:

```
Step 1. 모든 Input 노드의 connectionId → DB 서버 주소(host:port) 추출
Step 2. Output 노드의 connectionId → 실행 타겟 서버 주소 추출
Step 3. 각 노드 그룹이 동일 서버인지 판단

동일 서버 그룹 → SQL Pushdown 경로 (현재 방식 유지)
이기종 그룹    → Fetch-and-Process 경로 (PHASE 2 신규)
```

### 4-2. 실행 경로별 처리

```
┌─────────────────────────────────────────────────┐
│                   Job 실행 계획                   │
└──────────────┬──────────────────────────────────┘
               │ IR 분석
       ┌───────▼───────┐
       │  경로 라우터   │  ← PHASE 1 신규
       └──┬────────┬───┘
          │        │
    ┌─────▼──┐  ┌──▼──────────────┐
    │Pushdown│  │ Fetch-and-Process│  ← PHASE 2 신규
    │  경로  │  │      경로        │
    └─────┬──┘  └──┬──────────────┘
          │        │
    단일 SQL    소스별 Fetch
    실행(현재)  → 청크 처리
    (변경 없음) → 엔진 내 조인/변환
                → 타겟 write
```

## 5. 이기종 JOIN 전략 `[PHASE 2~3 설계 목표]`

### 5-1. 전략 선택 기준

```
두 Input의 예상 데이터 크기 비교 (통계 카탈로그 또는 캐시 활용):
  양쪽 모두 동일 서버
    → SQL Pushdown JOIN (현재 방식)
  한쪽이 기준치(예: 100만 건) 이하
    → Broadcast Join (소규모를 메모리에 올리고 대용량 측을 청크 스트리밍)
  양쪽 모두 대용량
    → Hash Join with Disk Spill (PHASE 3, 구현 복잡도 높음)
```

### 5-2. Broadcast Join (소규모 + 대용량) `[PHASE 2]`

```
소규모 테이블 (코드/마스터성): 전체 fetch → 메모리 해시맵 구성
대용량 테이블 (트랜잭션): 청크 단위 스트리밍 fetch

처리 흐름:
  [소규모 전체 fetch] → HashMap<key, Row>
  [대용량 청크 fetch] → 청크별로 HashMap 룩업 → 결과 타겟 write
  → 메모리 사용량 = 소규모 테이블 크기만큼만
```

예시: 고객 마스터(10만 건) JOIN 주문(1억 건)

```
고객 마스터 → HashMap (메모리, ~50MB)
주문 10만 건 청크 ─┐
주문 10만 건 청크  ├─ HashMap 룩업 → 타겟 누적 write
주문 10만 건 청크 ─┘
...
```

### 5-3. Hash Join with Disk Spill (대용량 + 대용량) `[PHASE 3 - 장기 과제]`

> **⚠️ 구현 복잡도 경고**
> JVM 내부에서 직접 구현 시 고려해야 할 사항:
> Spill 파일 포맷, 해시 충돌 처리, 파티션 재분배, 장애 복구, 임시 파일 정리 시점 등
> 상용 엔진(Spark, Flink)도 이 부분을 수년에 걸쳐 안정화했음.
> 공수 추정 "1~2주"는 PoC 수준이며, 운영 안정화까지는 훨씬 오래 걸린다.

```
양쪽 모두 청크 단위 fetch
파티션 해시로 디스크 임시 버킷 분산
버킷 단위로 조인 실행 → 타겟 write

→ 메모리 사용량 고정 (설정된 버퍼 크기)
→ 데이터 크기에 무관하게 안정적
→ 속도는 Broadcast보다 느림
→ 구현 복잡도는 이 문서의 모든 항목 중 최고 수준
```

### 5-4. 전략 자동 선택 — 크기 추정 방법 `[PHASE 2]`

> **⚠️ COUNT(*) 직접 실행 금지**
> 대형 테이블에서 매 실행마다 `COUNT(*)`는 그 자체로 무거운 풀스캔.
> 아래 우선순위로 크기 추정:

```
1순위: DB 통계 카탈로그 활용 (실행 비용 없음)
  PostgreSQL: pg_class.reltuples
  Oracle:     DBA_TABLES.NUM_ROWS (ANALYZE 후 갱신됨)
  MySQL/MariaDB: information_schema.TABLES.TABLE_ROWS (추정값)

2순위: 메타 테이블 캐시 활용
  etl_table_stats (job_id, node_id, estimated_rows, collected_at)
  → 주기적으로만 갱신 (예: 24시간마다)

3순위: 실제 COUNT(*) (소규모 테이블이 확실한 경우만 허용)
  기준: 마지막 캐시 기준으로 이미 소규모로 판정된 테이블
```

## 6. Chunk 스트리밍 처리 구조 `[PHASE 2 설계 목표]`

이기종 Fetch 경로의 핵심. 메모리 폭발을 방지하는 근간.

```
청크 크기: 설정 가능 (기본 10,000 rows)

처리 루프:
  last_id = 0
  while true:
    rows = fetch(query, last_id, chunkSize)  -- Keyset Pagination
    if rows.empty → break
    process(rows)       → 변환/조인
    write(rows, target) → 타겟 누적 write (UPSERT 권장)
    last_id = rows.last.id
```

**페이징 방식 비교:**

```sql
-- ❌ OFFSET 방식 (대용량에서 느려짐, 사용 지양)
SELECT * FROM orders LIMIT 10000 OFFSET 5000000
-- 이유: OFFSET이 클수록 앞의 행을 버리기 위해 모두 읽음

-- ✅ Keyset Pagination (PK 기반, 권장)
SELECT * FROM orders WHERE id > {last_id} ORDER BY id LIMIT 10000
-- PostgreSQL / MariaDB: 위 문법 그대로 사용

-- ✅ Oracle 12c 이상
SELECT * FROM orders WHERE id > {last_id} ORDER BY id
FETCH NEXT 10000 ROWS ONLY

-- ✅ Oracle 12c 미만 (서브쿼리 필요)
SELECT * FROM (
  SELECT t.*, ROWNUM rn
  FROM (SELECT * FROM orders WHERE id > {last_id} ORDER BY id) t
  WHERE ROWNUM <= 10000
)
-- 주의: Oracle에서 ROWNUM은 ORDER BY 전에 붙으므로
--       반드시 서브쿼리로 ORDER BY 먼저 적용 후 ROWNUM 사용
```

## 7. 구현 아키텍처 설계

### 7-1. 실행 엔진 구조 변경 `[PHASE 1~2 설계 목표]`

```
현재 (구현됨):
  SqlPushdownAdapter → SqlPushdownCompiler → 단일 SQL 실행

PHASE 1 목표:
  ExecutionRouter
    └── PushdownExecutor     (현재 방식, 동일 서버, 변경 없음)

PHASE 2 목표:
  ExecutionRouter
    ├── PushdownExecutor     (현재 방식)
    └── FetchAndProcessExecutor  (신규)
          ├── SourceFetcher        (소스별 청크 fetch)
          ├── JoinProcessor        (Broadcast Join)
          ├── TransformProcessor   (tMap 등 변환)
          └── TargetWriter         (타겟 청크 write)

PHASE 3 목표:
  FetchAndProcessExecutor에 HashJoinProcessor 추가
  (Disk Spill 포함)
```

### 7-2. 노드 IR 확장 (T_JDBC_INPUT) `[PHASE 1~2 설계 목표]`

> 현재 T_JDBC_INPUT은 `connectionId, tableName, query, columns`만 사용.
> 아래 필드는 전부 추가 예정 항목이며 현재 JobIR에 없음.

```json
{
  "type": "T_JDBC_INPUT",
  "config": {
    "connectionId": "...",
    "tableName": "orders",
    "incremental": {
      "enabled": true,
      "mode": "TIMESTAMP",
      "column": "updated_at",
      "watermarkVar": "last_run_orders"
    },
    "partition": {
      "enabled": false,
      "column": "order_date",
      "value": "${context.batch_date}"
    },
    "fetchHint": {
      "strategy": "AUTO"   // AUTO | BROADCAST | STREAMING
    }
  }
}
```

### 7-3. Watermark 관리 테이블 (메타 DB) `[PHASE 1 도입 예정]`

> 현재 Flyway 마이그레이션에 없음. PHASE 1 구현 시 마이그레이션 추가 필요.

```sql
CREATE TABLE etl_watermarks (
  job_id          UUID,
  node_id         VARCHAR(100),
  watermark_key   VARCHAR(100),
  watermark_value VARCHAR(255),
  updated_at      TIMESTAMP WITH TIME ZONE,  -- 타임존 명시 필수
  PRIMARY KEY (job_id, node_id, watermark_key)
);
```

- 실행 성공 시에만 갱신 → 실패 시 동일 범위 재처리 (멱등성 보장)
- `watermark_value`는 UTC 기준으로 저장

## 8. 구현 우선순위 (로드맵)

### PHASE 1 — 즉시 구현 가능 (단기)

```
[P1-1] Watermark 기반 증분 로딩
  - Input 노드 설정에 증분 옵션 추가 (JobIR 확장)
  - 컴파일러에서 WHERE 조건 자동 삽입
  - etl_watermarks 테이블 Flyway 마이그레이션
  - 실행 완료 후 watermark 갱신 (성공 시에만)
  - 경계값 처리 전략 결정 (>= + UPSERT 또는 > + 실행시작시각)
  - 예상 공수: 4~5일

[P1-2] ExecutionRouter 기반 분기 구조
  - 동일 서버 판단 로직 (host:port 비교)
  - Pushdown / Fetch 경로 분기 (Fetch 경로는 PHASE 2에서 구현)
  - 기존 Pushdown 경로는 현행 그대로 유지
  - 예상 공수: 2일
```

### PHASE 2 — 이기종 기본 지원 (중기)

```
[P2-1] FetchAndProcessExecutor 구현
  - Keyset Pagination 청크 스트리밍 fetch (DB별 페이징 문법 분기)
  - tMap/tFilter 등 변환 인메모리 적용
  - 타겟 청크 write (UPSERT 지원)
  - 예상 공수: 1~2주

[P2-2] Broadcast Join 구현
  - 테이블 통계 카탈로그 기반 크기 추정
  - etl_table_stats 캐시 테이블 도입
  - HashMap 기반 인메모리 Broadcast
  - 대용량 측 스트리밍 + 룩업
  - 예상 공수: 1~2주

[P2-3] 파티션 프루닝
  - Input 노드 파티션 설정 UI
  - 컴파일러/Fetcher에서 파티션 조건 적용
  - 예상 공수: 3~4일
```

### PHASE 3 — 고도화 (장기, 우선순위 재검토 필요)

```
[P3-1] Hash Join with Disk Spill
  - 대용량 + 대용량 이기종 JOIN
  - 임시 파일 기반 버킷 처리
  - Spill 파일 관리, 장애 복구, 정리 로직 포함
  - ⚠️ 공수 및 안정화 기간 사전 충분한 검토 필요
  - 예상 공수: 최소 1개월 이상 (운영 안정화 포함)

[P3-2] 실행 계획 자동 최적화
  - 통계 기반 전략 자동 선택 고도화
  - 예상 공수: 1~2주

[P3-3] CDC 연동 (별도 트랙)
  - Debezium 등 외부 CDC 툴 연동
  - 실시간 파이프라인 지원
  - 현재 배치 아키텍처와 완전히 분리된 별도 설계 필요
```

## 10. 리스크 분석

### 🔴 HIGH — 데이터 유실/오염 가능성

#### R1. Watermark 갱신과 타겟 Write의 원자성 파괴

```
시나리오:
  타겟 DB write 성공 → watermark 갱신 중 ETL 서버 장애
  → 다음 실행 시 watermark가 이전 값 → 동일 범위 재처리 → 중복 데이터 누적

더 위험한 시나리오:
  watermark 먼저 갱신 → 타겟 write 실패
  → 다음 실행 시 watermark가 이미 앞으로 넘어감 → 해당 구간 데이터 영구 유실
```

**대응:** 반드시 타겟 write 완전 성공 확인 후 watermark 갱신 순서 고정.
watermark 갱신 실패 시에도 재시도 가능하도록 멱등성 설계(UPSERT) 적용.

---

#### R2. Late Arriving Data (지연 데이터) — Watermark로 감지 불가

```
시나리오:
  배치가 어제 23:00 이후 데이터를 처리
  → 소스 시스템이 오늘 02:00에 어제 데이터를 재처리하며 updated_at = 어제 날짜로 삽입
  → watermark는 이미 오늘로 넘어감 → 해당 행 영구 유실
```

**실제 발생 케이스:**
- 외부 연동 데이터가 지연 수신 후 과거 날짜로 적재되는 경우
- 소스 DB에서 수동 정정 작업 시 updated_at을 원래 일자로 설정하는 경우
- 월말 마감 이후 소급 수정이 발생하는 업무

**대응 옵션:**
- 허용 지연 시간(lookback window) 설정: `WHERE updated_at >= last_run - INTERVAL '2 hours'`
- 단, lookback이 길수록 재처리 범위 증가 → 성능/멱등성 트레이드오프

---

#### R3. 물리 DELETE 감지 불가 — Watermark의 구조적 한계

Watermark 방식은 INSERT/UPDATE만 잡는다. 소스에서 행이 DELETE되면 타겟에 그대로 남는다.

```
소스: DELETE FROM orders WHERE id = 100
→ ETL 실행
→ 타겟: orders 테이블에 id=100 행이 여전히 존재 → 데이터 불일치
```

**영향 받는 케이스:**
- GDPR/개인정보 삭제 요청 처리
- 소스 시스템에서 논리삭제(is_deleted 플래그)가 아닌 물리삭제를 사용하는 경우

**대응 옵션:**
- 소스 테이블에 is_deleted 컬럼 + updated_at 갱신 규약 적용 (소스 시스템 협의 필요)
- 주기적 FULL SYNC 배치 병행 (예: 매주 1회 전체 비교/정합)
- 물리 DELETE가 많은 테이블은 CDC로만 처리 가능 (Watermark 한계)

---

### 🟠 MEDIUM — 운영 중 오동작 가능성

#### R4. Clock Skew (ETL 서버 ↔ 소스 DB 서버 시계 오차)

```
ETL 서버 시각: 10:00:00
소스 DB 시각:  10:00:30 (30초 빠름)

패턴 B 사용 시 (watermark = 실행 시작 시각):
  watermark = '10:00:00' (ETL 서버 기준으로 저장)
  소스 DB에서 10:00:00 ~ 10:00:30 사이 삽입된 행:
    → updated_at이 ETL watermark보다 앞서지만 실제로는 이후 데이터
    → 다음 배치까지 30초 치 데이터 처리 지연
    → 시계 오차가 크면(수 분 이상) 유실처럼 보임
```

**대응:** watermark 비교는 항상 소스 DB 서버 시각 기준으로 수행.
`SELECT NOW()` 등으로 소스 DB 현재 시각을 직접 읽어 watermark로 사용.

---

#### R5. updated_at 컬럼 신뢰성 문제

Watermark는 소스 테이블의 updated_at이 항상 정확하게 갱신된다는 전제가 있다.

**신뢰할 수 없는 케이스:**
- DBA가 운영 DB를 직접 UPDATE 하면서 updated_at을 갱신하지 않은 경우
- 소스 애플리케이션 버그로 일부 UPDATE가 updated_at을 갱신 안 하는 경우
- 소스 테이블이 SOFT DELETE 대신 물리 DELETE + INSERT 패턴을 사용하는 경우
- 소스 DB가 다른 팀 관리여서 컬럼 갱신 규약 통제 불가

**대응:** Watermark 적용 전 소스 테이블의 updated_at 신뢰성을 반드시 검증.
신뢰할 수 없는 테이블은 Watermark 대신 FULL SYNC 또는 CDC로만 처리.

---

#### R6. Keyset Pagination의 PK 전제 조건

Keyset Pagination은 **단조증가하는 단일 숫자 PK**를 전제로 한다.

**동작하지 않는 케이스:**
```
UUID(랜덤) PK:   WHERE id > 'a1b2c3...'  -- 정렬 기준 없음, 페이지 건너뜀 발생
복합 PK:         WHERE (user_id, order_id) > (?, ?)  -- DB별 문법 다름, 구현 복잡
문자열 PK:       WHERE code > 'ABC'  -- 문자 정렬 = 숫자 정렬 아님
updated_at 기준: 같은 타임스탬프 다수 존재 시 중복/누락
```

**대응:** 이기종 Fetch 경로 구현 시 소스 테이블의 PK 타입을 사전에 검증하는 로직 필요.
UUID PK 테이블은 OFFSET 방식(느리지만 정확) 또는 별도 surrogate key 활용.

---

#### R7. 청크 처리 중 소스 데이터 변경 (Snapshot 격리 없는 경우)

```
청크 1 fetch 완료: id 1 ~ 10000
소스 DB에서 id 5000 행이 UPDATE됨 (id 변경 없음, 다른 컬럼)
청크 2 fetch: id 10001 ~

→ 결과적으로 id 5000의 변경된 데이터가 청크 1에 반영됐는지 여부는
  fetch 타이밍에 따라 달라짐 → 배치 결과의 시점 일관성 없음
```

심각한 경우:
```
청크 1 fetch 중 소스에서 대량 DELETE 발생
→ 이후 청크의 OFFSET이 어긋남 (OFFSET 방식 사용 시)
→ 일부 행 누락
```

**대응 (선택):**
- 청크 전체를 하나의 트랜잭션(REPEATABLE READ)으로 fetch
  → 단, 장시간 스냅샷은 소스 DB undo/vacuum에 부하 (특히 PostgreSQL, Oracle)
- 허용 가능한 수준의 일관성 불일치로 설계 (일일 배치라면 대부분 허용 가능)

---

#### R8. Broadcast Join — 통계 오래됨으로 인한 OOM

```
시나리오:
  pg_class.reltuples = 80,000 (마지막 ANALYZE 기준)
  실제 현재 row count = 5,000,000 (데이터 폭증 후 ANALYZE 미실행)

  → 소규모 판정(80,000 < 100만 기준치) → Broadcast 경로 선택
  → 전체 fetch 시도 → 5백만 건 × row size → JVM OOM
```

**대응:**
- Broadcast 전 실제 fetch 시작 후 메모리 임계치 초과 시 즉시 중단하는 안전장치 추가
- 또는 통계 수집 주기를 강제하는 스케줄 도입 (ANALYZE 자동 실행)
- Broadcast 크기 상한(예: JVM 힙의 30%)을 하드코딩 제한으로 설정

---

#### R9. host:port 동일성 판단의 함정

ExecutionRouter가 같은 서버 여부를 `host:port` 문자열 비교로 판단할 경우:

```
같은 서버인데 다르게 표기된 경우 → 이기종으로 오판:
  Connection A: host = "localhost"
  Connection B: host = "127.0.0.1"

  Connection A: host = "db-server-01"
  Connection B: host = "10.0.0.15"  (같은 서버, 도메인/IP 차이)
```

**오판 결과:** 동일 서버인데 Fetch-and-Process 경로를 타게 됨 → 불필요한 데이터 이동 + 성능 저하.
Pushdown이 불가능하진 않지만 최적 경로를 놓치는 것.

**대응:** 판단 로직에 도메인 → IP 정규화 추가.
또는 connectionId가 같으면 무조건 동일 서버로 판단하는 단순화 방식.

### 🟡 LOW — 설계 시 사전 고려 필요

#### R10. 소스 DB 부하 — OLTP 운영 DB에서 대량 Fetch

이기종 Fetch 경로에서 ETL이 소스 OLTP DB에 반복적으로 대용량 쿼리를 날리면
운영 서비스 쿼리와 경쟁하여 소스 DB 성능에 영향.

**대응:**
- Fetch 속도 제한(throttling): 청크 사이에 sleep 추가 또는 fetch 속도 상한 설정
- 소스 DB Read Replica 사용 강력 권장 (운영 Master 직접 접근 지양)
- 소스 DB 접근 가능 시간대 제한 (예: 새벽 배치 전용)

---

#### R11. Schema Evolution (소스 테이블 구조 변경)

소스 테이블의 컬럼이 추가/삭제/타입 변경될 경우:

```
컬럼 추가: CTE의 SELECT *는 자동 포함 → 타겟 테이블에 해당 컬럼 없으면 에러
컬럼 삭제: tMap에서 해당 컬럼 참조 시 런타임 에러
타입 변경: 암묵적 캐스팅 실패 가능 (VARCHAR → INT 등)
updated_at 컬럼 삭제: Watermark 증분 즉시 중단
```

**대응:** 실행 전 소스 컬럼 메타데이터와 IR에 저장된 컬럼 목록 비교하는 validation 추가.
변경 감지 시 즉시 실패 처리 + 알림 → 침묵 실패 방지.

## 12. 핵심 결론

1. **FULL SCAN 문제는 Watermark 증분이 가장 빠른 해결책** — CDC보다 먼저, 효과는 즉각적. 단, 경계값(> vs >=) 처리와 타임존 통일은 초기 설계 시 반드시 결정.
2. **이기종 JOIN은 크기 기반 전략 선택이 핵심** — 무조건 Hash Join 하면 소규모 케이스에서 오히려 느리고, COUNT(*) 직접 실행보다 통계 카탈로그 활용이 현실적.
3. **Pushdown 경로는 건드리지 않음** — 현재 동일 DB 환경에서 이미 최적, 기존 동작 보장.
4. **청크 스트리밍은 선택이 아닌 필수** — 이기종 fetch를 전체 메모리 로드로 구현하면 운영 중 반드시 OOM 발생.
5. **Hash Join + Disk Spill은 PHASE 3 장기 과제** — 상용 엔진도 수년에 걸쳐 안정화. 충분한 공수와 검토 후 착수.
6. **CDC는 별도 트랙** — 현재 배치 아키텍처와 실시간 아키텍처는 설계 철학이 다름, 섣불리 혼합하면 복잡도만 증가.

## 13. 작업 Plan

### 사전 결정 사항 (구현 시작 전 반드시 확정)

아래 항목은 한번 결정되면 이후 DB 마이그레이션/IR 포맷 변경이 동반되므로, 착수 전 명시적으로 결정해야 한다.

| 결정 항목 | 옵션 A | 옵션 B | 권장 |
|----------|--------|--------|------|
| Watermark 경계값 전략 | `>=` + 타겟 UPSERT (멱등성 보장) | `>` + 실행 시작 시각 저장 | **옵션 A** (UPSERT와 세트로 설계) |
| 동시 실행 Job 잠금 방식 | DB 레벨 Advisory Lock (PostgreSQL) | 메타 테이블 상태 컬럼 + CAS UPDATE | **옵션 B** (이기종 DB 환경 대응) |
| tMap 인메모리 변환 방식 | SQL 컴파일러 재사용 (CTE 생성) | Row-level 프로세서 (Kotlin 코드로 변환) | **옵션 B** (Fetch 경로는 행 단위 처리) |
| 타겟 UPSERT 방언 분기 | 단일 MERGE INTO 문 (표준) | DB 타입별 분기 구현 | **옵션 B** (DB별 지원 범위 다름) |

**동시 실행 Job 잠금 필요 이유:**
같은 Job이 스케줄러에 의해 동시에 두 번 실행되면 watermark가 같은 값을 읽은 채로 두 번 처리 → 타겟에 중복 적재. 반드시 같은 Job ID의 실행은 직렬화해야 함.

**타겟 UPSERT 방언 비교:**
```sql
-- PostgreSQL (ON CONFLICT)
INSERT INTO target (...) VALUES (...)
ON CONFLICT (pk_col) DO UPDATE SET col1 = EXCLUDED.col1, ...

-- Oracle (MERGE INTO)
MERGE INTO target t
USING (SELECT ... FROM DUAL) s ON (t.pk_col = s.pk_col)
WHEN MATCHED THEN UPDATE SET t.col1 = s.col1
WHEN NOT MATCHED THEN INSERT (...) VALUES (...)

-- MySQL / MariaDB (ON DUPLICATE KEY UPDATE)
INSERT INTO target (...) VALUES (...)
ON DUPLICATE KEY UPDATE col1 = VALUES(col1), ...
```

### PHASE 1 — Watermark 증분 로딩 + ExecutionRouter 분기

**목표:** FULL SCAN 제거. 이기종 실행 경로 분기 구조만 만들고 Fetch 경로는 PHASE 2에서 채움.

#### 1단계: 메타 DB 스키마 확장

파일: `backend/src/main/resources/db/migration/`

### PHASE 2 — 이기종 DB 기본 지원

**목표:** 다른 서버의 DB 간 데이터 이동 및 Broadcast Join 구현.

#### 1단계: FetchAndProcessExecutor 골격

파일: `backend/src/main/kotlin/com/platform/etl/execution/FetchAndProcessExecutor.kt` (신규)

```
책임 분리:
  SourceFetcher       → DB별 Keyset Pagination 청크 fetch
  RowTransformer      → tMap 컬럼 매핑 + Expression 적용 (행 단위)
  TargetWriter        → 청크 단위 UPSERT (DB 방언 분기)
```

#### 2단계: SourceFetcher — DB별 Keyset Pagination

파일: `backend/src/main/kotlin/com/platform/etl/execution/SourceFetcher.kt` (신규)

```kotlin
interface SourceFetcher {
    fun fetchChunks(connection: Connection, query: String, pkCol: String,
                    chunkSize: Int): Sequence<List<Row>>
}
```

DB 방언 분기:
- PostgreSQL / MariaDB: `WHERE {pk} > ? ORDER BY {pk} LIMIT ?`
- Oracle 12c+: `WHERE {pk} > ? ORDER BY {pk} FETCH NEXT ? ROWS ONLY`
- Oracle 12c-: 서브쿼리 + ROWNUM 방식

UUID PK 감지 시 경고 로그 + OFFSET 방식 폴백.

#### 3단계: RowTransformer — tMap 인메모리 적용

파일: `backend/src/main/kotlin/com/platform/etl/execution/RowTransformer.kt` (신규)

- IR의 T_MAP 노드에서 컬럼 매핑 정의 읽기
- Expression 문자열 파싱 → 행 단위 적용
  - `TRIM(col)`: 공백 제거
  - `UPPER(TRIM(col))`: 대문자 + 공백 제거
  - `COALESCE(col, 0)`: null → 0
  - `CAST(col AS DATE)`: 타입 변환
- 복잡한 Expression(서브쿼리, 함수 중첩 등)은 초기엔 그대로 통과 후 경고

#### 4단계: TargetWriter — UPSERT 방언 분기

파일: `backend/src/main/kotlin/com/platform/etl/execution/TargetWriter.kt` (신규)


#### 5단계: Broadcast Join 구현

파일: `backend/src/main/kotlin/com/platform/etl/execution/BroadcastJoinHelper.kt` (신규)

```kotlin
class BroadcastJoinHelper(private val maxMemoryBytes: Long = 512 * 1024 * 1024) {
    fun buildHashMap(rows: List<Row>, keyCol: String): HashMap<Any?, Row>
    fun lookup(map: HashMap<Any?, Row>, keyVal: Any?): Row?
}
```

- 소규모 테이블 전체 fetch → HashMap 구성
- fetch 중 누적 바이트 추정 → `maxMemoryBytes` 초과 시 `BroadcastOOMException` (안전장치, R8 대응)
- 대용량 측 청크 스트리밍 → 각 행마다 HashMap.lookup() 수행

#### 6단계: 크기 추정 — 통계 카탈로그 조회

파일: `backend/src/main/kotlin/com/platform/etl/execution/TableStatsService.kt` (신규)

```kotlin
fun estimateRowCount(conn: Connection, schema: String, table: String, dbType: DbType): Long
```

DB별 통계 카탈로그 쿼리:
- PostgreSQL: `SELECT reltuples FROM pg_class WHERE relname = ?`
- Oracle: `SELECT NUM_ROWS FROM DBA_TABLES WHERE TABLE_NAME = UPPER(?)`
- MySQL: `SELECT TABLE_ROWS FROM information_schema.TABLES WHERE TABLE_NAME = ?`

null/0 반환 시 `etl_table_stats` 캐시 조회 → 없으면 Broadcast 경로 포기 + 경고 로그.

**PHASE 2 완료 기준:**
- PostgreSQL A 서버 → PostgreSQL B 서버 데이터 이동 동작 검증
- Oracle → PostgreSQL Broadcast Join 검증 (마스터 테이블 < 100만 건)
- UPSERT writeMode로 Watermark + 중복 없이 누적 적재 검증

| 리스크 | 대응 작업 | 포함 Phase |
|--------|----------|-----------|
| R1 (Watermark 원자성) | write 완전 성공 후 watermark 갱신 순서 고정, try-finally | PHASE 1 |
| R2 (Late Arriving) | lookback window 설정 옵션 추가 (Input 노드 UI) | PHASE 1 |
| R3 (물리 DELETE) | 문서화 + 주기적 FULL SYNC 배치 가이드 제공 | 설계 결정 |
| R4 (Clock Skew) | watermark 기준값을 소스 DB `NOW()` 로 수집 | PHASE 1 |
| R5 (updated_at 신뢰성) | Watermark 활성화 전 컬럼 존재 검증 | PHASE 1 |
| R6 (UUID PK) | PK 타입 사전 검증 + OFFSET 폴백 | PHASE 2 |
| R7 (Snapshot 격리) | 허용 범위 합의 후 문서화; 필요 시 REPEATABLE READ 옵션 | PHASE 2 |
| R8 (Broadcast OOM) | BroadcastJoinHelper 메모리 상한 안전장치 | PHASE 2 |
| R9 (host:port 오판) | connectionId 동일성 우선 비교 로직 | PHASE 1 |
| R10 (소스 DB 부하) | 청크 사이 throttle 설정 옵션 추가 | PHASE 2 |
| R11 (Schema Evolution) | 실행 전 IR 컬럼 목록 vs 소스 메타데이터 검증 | PHASE 2 |
