# Transaction Control 기능 분석 및 구현 방향

## 1. 파일 분석 (기능제안.md)

### 1.1 제안 내용 요약

- **Transaction Control 컴포넌트** 추가: SQL 실행과 **COMMIT / ROLLBACK**을 Job 설계 단에서 제어할 수 있는 장치.
- **Preview Mode와의 관계**: 대체가 아니라 **보완**. Preview = 개발/테스트 시 결과 검증, Commit/Rollback = 운영 시 데이터 안정성.
- **필요 시나리오**: ① Multi-table Load, ② SCD 처리, ③ Batch Aggregation, ④ CDC Merge, ⑤ 에러 시 복구.
- **제안 구조**:
  - `[Job Start]` → **BEGIN TRANSACTION** → Step1 → Step2 → Step3 → **Commit 컴포넌트**
  - 에러 발생 시 → **Rollback 컴포넌트**
  - **Preview Mode일 때는 BEGIN/COMMIT/ROLLBACK 모두 비활성화(Disable)**.

### 1.2 타당성

- **타당함.** 상용 ETL(Talend tDBCommit/tDBRollback, Informatica Transaction Control, DataStage Commit stage)에서 표준으로 제공하는 기능이며, Plan_ETL.md 1.5절에서도 DB 컴포넌트 예시로 "Commit/Rollback"이 이미 명시되어 있음.
- 기능제안.md의 "고객 INSERT → 주문 INSERT → 결제 INSERT 중 결제 실패 시, Rollback 없으면 데이터 깨짐" 예시는 현재 구현의 한계와 정확히 일치함(아래 2절 참고).

## 2. 필요성 확인 및 현재 구조 한계

### 2.1 현재 실행 구조 (코드 기준)

- **Preview Mode** (ExecutionService.kt, SqlPushdownAdapter.kt 177~180행): `previewMode=true`이면 **Output 노드만 SKIP** (SQL 실행 없이 "[Preview Mode: Output 건너뜀]" 반환). Input/변환 로직은 실행되나 DB에 쓰지 않음 → **개발/테스트용으로 적합**.
- **Execute Mode**:
  - **Input 노드**: 매 노드마다 `DriverManager.getConnection().use`로 **새 연결** → COUNT 쿼리만 수행 후 연결 종료. 트랜잭션 없음.
  - **Output 노드**: 역시 **노드마다 새 연결**, `autoCommit = false` → 해당 노드의 INSERT(및 TRUNCATE)만 수행 후 `commit()` 또는 예외 시 `rollback()`.
  - 따라서 **노드 단위 트랜잭션만 존재**하며, **Job 단위 또는 여러 Output을 묶는 단일 트랜잭션은 없음**.

### 2.2 현재 구조에서의 문제 (기능제안.md 예시와 동일)

- **다중 테이블 적재** 시: Output 노드 3개(고객/주문/결제)면 → 노드1 성공 시 **commit** → 노드2 성공 시 **commit** → 노드3 실패 시 노드1·2는 이미 커밋되어 **롤백 불가** → 데이터 불일치.
- **컴파일러 제약**: SqlPushdownCompiler.kt 42행은 단일 T_JDBC_OUTPUT만 가정. Multi-output 파이프라인은 현재 미지원이지만, 향후 Multi-output 지원 시에도 **트랜잭션 경계 없으면** 동일 문제 발생.
- **테스트/운영**: 큰 Job이나 많은 노드 구성에서 "일부만 반영된 상태"를 피하려면 **명시적 Commit/Rollback**으로 원자성 구간을 정의할 수 있어야 함.

### 2.3 필요성 정리

| 관점              | 내용                                                                                                                              |
| ----------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **운영 안정성**   | Multi-table Load, SCD, CDC Merge 등에서 **전부 성공 시에만 반영, 실패 시 전부 롤백**이 필수.                                      |
| **테스트**        | Preview로 로직 검증 후, Execute 시에도 **트랜잭션 경계를 Job 설계에서 제어**할 수 있어야 대규모/복잡 Job 테스트가 안전함.         |
| **상용 ETL 정렬** | Talend/Informatica/DataStage와 동일한 설계 철학으로, "Preview + Transaction Control" 조합은 차별화 포인트이자 표준 요구사항 충족. |

## 3. 상용 ETL vs 현재 프로그램 차이 및 방안

### 3.1 비교 요약

| 항목                    | 상용 ETL (Talend/Informatica/DataStage)                                                                     | 현재 ETL 플랫폼                                                         |
| ----------------------- | ----------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **트랜잭션 단위**       | Job 또는 사용자 지정 구간(BEGIN ~ COMMIT). 여러 Output이 한 트랜잭션에 포함 가능.                           | Output **노드마다** 별도 Connection, 노드 단위 commit/rollback만 존재.  |
| **Commit/Rollback**     | **컴포넌트**로 그래프 상에 명시(tDBCommit, tDBRollback 등). 실행 시 해당 지점에서 commit()/rollback() 호출. | 컴포넌트 없음. 각 Output 노드 내부에서만 commit/rollback.               |
| **Connection 생명주기** | Job(또는 Subjob) 단위로 Connection 유지 후, Commit/Rollback 컴포넌트에서 제어.                              | 노드 실행 시마다 `getConnection().use`로 **즉시 반환**. 구간 공유 없음. |
| **Preview / Execute**   | 디버그/Preview 모드와 실제 실행 모드 분리.                                                                  | **Preview Mode = Output 스킵.** Execute = 노드 단위 트랜잭션.           |

### 3.2 구현 방향 (적절한 방안)

**목표**: 기능제안.md 구조에 맞게 **Commit/Rollback 컴포넌트**를 도입하고, **선택한 구간을 하나의 트랜잭션**으로 실행.

**핵심 변경**:

1. **트랜잭션 경계 정의**
   - IR에 **T_COMMIT**, **T_ROLLBACK**(또는 T_DB_COMMIT, T_DB_ROLLBACK) 타입 추가 (JobIR.kt ComponentType).
   - 실행 순서: 위상 정렬된 노드 순으로 진행. "Job Start 시 BEGIN, T_COMMIT에서 commit, T_ROLLBACK 또는 예외 시 rollback" 단순 모델부터 적용.

2. **Connection 공유**
   - "BEGIN → ... → COMMIT/ROLLBACK" 구간 동안 **한 Connection을 유지**하고, 해당 구간 내 Input/Output 노드가 이 Connection을 사용하도록 변경.
   - 구현 옵션: (A) ExecutionPlan 또는 Adapter에서 "현재 트랜잭션 구간용 Connection" 캐시. (B) Job 단위로 하나의 트랜잭션 Connection을 열고, Commit/Rollback 노드에서만 commit/rollback 호출.

3. **실행 시맨틱 (Preview vs Execute)**
   - **Preview Mode** (기능제안.md 5·6절 준수):
     - **BEGIN/COMMIT/ROLLBACK 컴포넌트는 아무 동작도 하지 않음(No-op).**
     - Output 노드는 기존처럼 **스킵** (SQL 실행 없음).
     - 즉, "BEGIN/COMMIT/ROLLBACK 모두 Disable"과 일치.
   - **Execute Mode**:
     - 트랜잭션 구간 시작(Job 시작 또는 직전 COMMIT 이후): `connection.setAutoCommit(false)`.
     - **T_COMMIT** 노드 도달: `connection.commit()`.
     - **T_ROLLBACK** 노드 도달 또는 구간 내 예외: `connection.rollback()`.
   - 에러 시: 해당 구간 rollback 후, ExecutionResult에 FAILED 반환(현재와 동일).

4. **컴파일러/다중 Output**
   - 현재 컴파일러는 **단일 Output** 전제. Transaction Control 도입 시 **먼저 단일 트랜잭션 구간(단일 Connection)** 으로 Commit/Rollback만 구현하고, Multi-output은 이후 단계로 확장.

**Preview와의 관계 정리**:

- Preview Mode일 때: Commit/Rollback 노드는 **실행만 스킵**(또는 SUCCESS지만 실제 commit/rollback 호출 안 함). "BEGIN/COMMIT/ROLLBACK 모두 Disable"과 일치.
- Execute Mode일 때만: 트랜잭션 경계와 Commit/Rollback 컴포넌트가 동작.

## 4. 정리 및 권장 순서

- **기능제안.md 내용은 타당하며**, 현재 구조(Preview만 있고 노드 단위 트랜잭션만 존재)로는 Multi-table/SCD/CDC 등에서 원자성을 보장할 수 없어, **Transaction Control 도입이 필요**함.
- **상용 ETL과의 차이**는 "Job(또는 구간) 단위 트랜잭션 + 명시적 Commit/Rollback 컴포넌트" 유무이며, 이를 **컴포넌트 추가 + Connection 생명주기/공유 변경**으로 맞추는 방안이 적절함.
- **권장 구현 순서**:
  1. **IR/팔레트**: `T_DB_COMMIT`, `T_DB_ROLLBACK` (또는 T_COMMIT, T_ROLLBACK) 추가.
  2. **실행 엔진**: 단일 Connection 기준으로 Job 전체를 한 트랜잭션 구간으로 두고, Commit 노드에서 commit(), Rollback 노드 또는 예외 시 rollback() 호출. (우선 **단일 Output** 가정 유지.)
  3. **Connection 공유**: 동일 connectionId를 쓰는 Input/Output 노드가 **같은 Connection 인스턴스**를 쓰도록 Adapter에서 Connection 캐시/인젝션.
  4. **Preview**: Commit/Rollback 노드는 previewMode일 때 No-op 또는 SKIPPED 처리.
  5. (선택) Multi-output 지원 시: Output별 컴파일 + 동일 Connection 구간만 한 트랜잭션으로 묶는 정책 확장.

이 순서로 진행하면 기능제안.md의 목적(Preview와 보완 관계, 운영 안정성, 상용 ETL 수준 설계)을 만족하면서도 현재 코드베이스 변경 범위를 제어할 수 있음.
