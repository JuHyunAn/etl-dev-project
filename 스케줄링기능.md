다만, 설계 단계에서 세 가지 결정을 먼저 내려야 합니다.

1. 스케줄의 소유권: Job 종속 vs 독립 엔티티 이게 가장 중요한 아키텍처 결정입니다.

- Option A — Job에 종속 (1:1)
jobs 테이블에 cron_expression, schedule_enabled 컬럼 추가
Talend 방식입니다. 단순하지만, 하나의 스케줄이 여러 Job을 순차 실행하는 파이프라인 구성이 불가능합니다.

- Option B — 독립 엔티티 (1:N, 권장)
schedules 테이블 → schedule_jobs (조인) → jobs
Airflow DAG 방식입니다. 하나의 스케줄이 여러 Job을 순서대로 실행할 수 있고, 의존 관계 표현이 가능합니다.
JUHYEON이 말한 **"연관 스케줄러끼리의 파이프라인 연결선"**을 구현하려면 Option B가 필수입니다.

권장하는 데이터 모델(참고용):

- 스케줄 정의 SQL
CREATE TABLE schedules (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            VARCHAR(200) NOT NULL,
    description     TEXT,
    cron_expression VARCHAR(100) NOT NULL,
    timezone        VARCHAR(50) DEFAULT 'Asia/Seoul',
    enabled         BOOLEAN DEFAULT false,
    
    -- Quartz 연동
    quartz_job_key    VARCHAR(200),
    quartz_trigger_key VARCHAR(200),
    
    -- 운영 정보
    last_fired_at   TIMESTAMP,
    next_fire_at    TIMESTAMP,
    consecutive_failures INT DEFAULT 0,
    
    -- 알림 설정
    alert_on_failure  BOOLEAN DEFAULT true,
    alert_channel     VARCHAR(50),  -- email, slack, webhook 등
    
    created_by      UUID REFERENCES users(id),
    created_at      TIMESTAMP DEFAULT now(),
    updated_at      TIMESTAMP DEFAULT now()
);

-- 스케줄 내 Job 실행 순서 (파이프라인)
CREATE TABLE schedule_steps (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    schedule_id     UUID NOT NULL REFERENCES schedules(id) ON DELETE CASCADE,
    job_id          UUID NOT NULL REFERENCES jobs(id),
    step_order      INT NOT NULL,           -- 실행 순서
    
    -- 의존 관계 (선행 step 완료 후 실행)
    depends_on_step_id UUID REFERENCES schedule_steps(id),
    
    -- 실행 조건
    run_condition   VARCHAR(20) DEFAULT 'ON_SUCCESS',  
    -- ON_SUCCESS: 선행 성공 시만 / ON_COMPLETE: 성공/실패 무관 / ON_FAILURE: 선행 실패 시만
    
    -- step별 override 옵션
    timeout_seconds INT DEFAULT 3600,
    retry_count     INT DEFAULT 0,
    retry_delay_seconds INT DEFAULT 60,
    context_overrides JSONB DEFAULT '{}',   -- Job context 변수 오버라이드
    
    enabled         BOOLEAN DEFAULT true,
    
    UNIQUE(schedule_id, step_order)
);

-- 스케줄 실행 이력 (로드맵 1번 executions와 연결)
CREATE TABLE schedule_executions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    schedule_id     UUID NOT NULL REFERENCES schedules(id),
    
    status          VARCHAR(20) NOT NULL,   -- RUNNING, SUCCESS, FAILED, PARTIAL, CANCELLED
    started_at      TIMESTAMP NOT NULL DEFAULT now(),
    finished_at     TIMESTAMP,
    
    -- 요약
    total_steps     INT,
    completed_steps INT DEFAULT 0,
    failed_steps    INT DEFAULT 0,
    skipped_steps   INT DEFAULT 0,
    
    trigger_type    VARCHAR(20) DEFAULT 'CRON',  -- CRON, MANUAL, API, RETRY
    error_summary   TEXT
);

-- step별 실행 이력 (기존 executions 테이블과 FK)
CREATE TABLE schedule_step_executions (
    id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    schedule_execution_id UUID NOT NULL REFERENCES schedule_executions(id) ON DELETE CASCADE,
    schedule_step_id      UUID NOT NULL REFERENCES schedule_steps(id),
    execution_id          UUID REFERENCES executions(id),  -- 기존 Job 실행 이력 연결
    
    status          VARCHAR(20) NOT NULL,
    started_at      TIMESTAMP,
    finished_at     TIMESTAMP,
    retry_attempt   INT DEFAULT 0,
    error_message   TEXT
);
```

이 구조의 핵심은 **schedule_steps.depends_on_step_id**입니다. 이걸로 step 간 의존 관계를 DAG로 표현할 수 있고, 프론트엔드에서 연결선으로 시각화할 수 있습니다.

## 3. 스케줄러 탭 — Airflow/GitHub Actions 벤치마킹

여기가 핵심입니다. 제가 벤치마킹 대상별로 **가져올 것과 버릴 것**을 정리합니다.

### Airflow에서 가져올 것
- **DAG 목록 뷰**: 스케줄 이름, cron, 최근 N회 실행 상태 (초록/빨강 원형 아이콘 나열) — 이건 한눈에 건강 상태가 보여서 매우 효과적입니다
- **Gantt 차트 뷰**: 스케줄 실행 시 각 step이 언제 시작/종료했는지 타임라인으로 표시
- **Tree 뷰 (실행 이력 매트릭스)**: 행=step, 열=실행 회차, 셀=상태 색상

### Airflow에서 버릴 것
- Python DAG 코드 기반 설정 (JUHYEON의 플랫폼은 GUI 기반이므로 불필요)
- XCom (step 간 데이터 전달) — 현 단계에서는 과도함

### GitHub Actions에서 가져올 것
- **실행 로그의 step별 접기/펼치기 UI** — 이게 정말 깔끔합니다
- **실행 시간 표시 바** (각 step 옆에 `2m 34s` 같은 duration 표시)
- **Manual trigger with inputs** — 수동 실행 시 context 변수 오버라이드 입력 폼

## 4. Quartz 연동 구조에 대한 의견

Quartz는 맞는 선택이지만, **Quartz를 얇게 쓰는 것**을 강력 권장합니다.
```
[schedules 테이블] ← 단일 진실의 원천 (Single Source of Truth)
        │
        ▼
[ScheduleService] ── 스케줄 CRUD + Quartz 동기화
        │
        ▼
[Quartz Scheduler] ── 순수하게 "언제 트리거할 것인가"만 담당
        │
        ▼
[ScheduleExecutionService] ── step 순서대로 Job 실행 오케스트레이션
        │
        ▼
[ExecutionService.execute()] ── 기존 Job 실행 로직 재사용

핵심 원칙은 Quartz는 타이머 역할만 하고, 실행 오케스트레이션·이력 관리·상태 추적은 전부 우리 코드에서 하는 것입니다. Quartz의 JobStore에 비즈니스 로직을 넣으면 나중에 디버깅이 극도로 어려워집니다. 

** Airflow가 성공한 이유 중 하나도 스케줄러와 실행기를 분리한 아키텍처 때문입니다.

kotlin// Quartz Job은 이것만 한다.

class ScheduleTriggerJob : Job {
    override fun execute(context: JobExecutionContext) {
        val scheduleId = context.mergedJobDataMap.getString("scheduleId")
        // 우리 서비스 호출만 하고 끝
        scheduleExecutionService.trigger(UUID.fromString(scheduleId))
    }
}

로드맵에서도 짚었지만 다시 강조합니다. 스케줄러 개발 전에 반드시:

실행 이력 저장 (로드맵 1번) — executions 엔티티/Repository/저장 로직 완성
HikariCP 풀링 (로드맵 3번) — 스케줄러가 cron마다 Job을 실행하면 DriverManager 직접 호출로는 커넥션 폭증 위험

이 두 가지가 없으면 스케줄러를 만들어도 실행 결과를 추적할 수 없고, 동시 실행 시 DB 커넥션이 고갈됩니다.
